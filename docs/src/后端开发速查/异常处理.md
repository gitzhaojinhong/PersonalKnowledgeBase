

## 1.1 概述

Controller 层抛出未捕获的异常时，SpringBoot 提供的异常处理机制会自动生效。

统一异常处理的价值：
- 统一错误响应格式，便于前端解析
- 避免技术性错误信息暴露给用户，提升体验
- 无需在每个方法中重复 try-catch，业务代码更简洁
- 控制异常信息输出，防止敏感信息泄露

SpringBoot 支持自适应响应：根据请求头 `Accept` 字段自动决定返回 JSON 还是 HTML 错误页，实现客户端设备自适应。

---

## 1.2 异常处理机制优先级

SpringMVC 的异常处理方案优先级高于 SpringBoot 默认方案。

流程如下：

```
Controller 抛出异常
       ↓
SpringMVC 是否有对应处理方案？
  ├── 有 → 执行 SpringMVC 处理（@ExceptionHandler / @ControllerAdvice）
  └── 无 → 启动 SpringBoot 默认错误处理（/error 端点）
```

## 1.3 前后端分离项目异常处理方案

前后端分离项目统一使用 SpringMVC 的异常处理方案，所有异常以 JSON 格式响应，前端接收后自行处理展示逻辑。

### 1.3.1 局部异常处理

在 Controller 类中定义带 `@ExceptionHandler` 的方法，只对当前 Controller 生效。

```java
@RestController
public class UserController {

    @GetMapping("/resource/{id}")
    public String getResource(@PathVariable Long id) {
        if (id == 1) {
            throw new IllegalArgumentException("无效ID：" + id);
        }
        return "ID = " + id;
    }

    // 只对当前 Controller 内的 IllegalArgumentException 生效
    @ExceptionHandler(IllegalArgumentException.class)
    public String handler(IllegalArgumentException e) {
        return "错误信息：" + e.getMessage();
    }
}
```

适用场景：某个 Controller 有特殊的异常处理逻辑，与全局处理不同。

### 1.3.2 全局异常处理

将异常处理方法提取到独立类，对所有 Controller 全局生效。

两种写法等价：
- `@ControllerAdvice` + `@ResponseBody` + `@ExceptionHandler`
- `@RestControllerAdvice` + `@ExceptionHandler`（推荐，更简洁）

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(IllegalArgumentException.class)
    public String handler(IllegalArgumentException e) {
        return "错误信息：" + e.getMessage();
    }
}
```

### 1.3.3 统一返回对象 Result

定义统一的响应结构，确保所有接口（含异常响应）格式一致：

```java
@Data
public class Result<T> {
    private int code;    // 业务状态码
    private String msg;  // 提示信息
    private T data;      // 数据体

    // 成功（带数据）
    public static <T> Result<T> success(T data) {
        Result<T> result = new Result<>();
        result.setCode(200);
        result.setMsg("操作成功");
        result.setData(data);
        return result;
    }

    // 成功（无数据）
    public static <T> Result<T> success() {
        return success(null);
    }

    // 失败
    public static <T> Result<T> error(int code, String msg) {
        Result<T> result = new Result<>();
        result.setCode(code);
        result.setMsg(msg);
        return result;
    }

    // 失败（带额外数据，如校验错误列表）
    public static <T> Result<T> error(int code, String msg, T data) {
        Result<T> result = new Result<>();
        result.setCode(code);
        result.setMsg(msg);
        result.setData(data);
        return result;
    }
}
```

### 1.3.4 自定义业务异常体系

基础异常类，所有自定义业务异常继承此类：

```java
public class BaseException extends RuntimeException {
    private int code;

    public BaseException(int code, String message) {
        super(message);
        this.code = code;
    }

    public BaseException(int code, String message, Throwable cause) {
        super(message, cause);
        this.code = code;
    }

    public int getCode() {
        return code;
    }
}
```

具体业务异常，按业务模块拆分，继承 BaseException（错误码常量也可用枚举类统一管理）：

```java
public class UserException extends BaseException {
    public static final int USER_NOT_FOUND = 1001;
    public static final int USER_DISABLED  = 1002;

    public UserException(int code, String message) {
        super(code, message);
    }
}
```

### 1.3.5 全局异常处理器完整实现

```java
@Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {

    /**
     * 自定义业务异常（BaseException 及其子类）
     */
    @ExceptionHandler(BaseException.class)
    public Result<Void> handleBaseException(BaseException ex, HttpServletRequest request) {
        log.warn("[业务异常] 请求路径: {}, 错误码: {}, 异常信息: {}",
                request.getRequestURI(), ex.getCode(), ex.getMessage());
        return Result.error(ex.getCode(), ex.getMessage());
    }

    /**
     * 参数校验异常（@Valid / @Validated 作用于 RequestBody）
     */
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public Result<List<String>> handleMethodArgumentNotValidException(MethodArgumentNotValidException ex) {
        List<String> errors = ex.getBindingResult().getFieldErrors()
                .stream().map(FieldError::getDefaultMessage).collect(Collectors.toList());
        log.warn("[参数校验异常] - {}", errors);
        return Result.error(400, "参数校验失败", errors);
    }

    /**
     * 参数校验异常（@Validated 直接校验方法参数，如 @PathVariable、@RequestParam）
     */
    @ExceptionHandler(ConstraintViolationException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public Result<List<String>> handleConstraintViolationException(ConstraintViolationException ex) {
        List<String> errors = ex.getConstraintViolations()
                .stream().map(v -> v.getPropertyPath() + ": " + v.getMessage())
                .collect(Collectors.toList());
        log.warn("[参数校验异常] - {}", errors);
        return Result.error(400, "参数校验失败", errors);
    }

    /**
     * 数据绑定异常（类型转换错误等）
     */
    @ExceptionHandler(BindException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public Result<List<String>> handleBindException(BindException ex) {
        List<String> errors = ex.getFieldErrors()
                .stream().map(FieldError::getDefaultMessage).collect(Collectors.toList());
        log.warn("[数据绑定异常] - {}", errors);
        return Result.error(400, "数据绑定错误", errors);
    }

    /**
     * 兜底处理：所有未被上述方法捕获的异常
     */
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public Result<Void> handleException(Exception ex, HttpServletRequest request) {
        log.error("[系统异常] 请求路径: {}，异常信息：", request.getRequestURI(), ex);
        return Result.error(500, "系统繁忙，请稍后再试");
    }
}
```

### 1.3.6 各层代码示例

Service 层清晰抛出业务异常：

```java
@Service
public class UserServiceImpl implements UserService {

    @Autowired
    private UserMapper userMapper;

    @Override
    public UserDTO getUserById(Long userId) {
        UserDTO user = userMapper.selectById(userId);
        if (user == null) {
            throw new UserException(UserException.USER_NOT_FOUND, "用户ID: " + userId + " 不存在");
        }
        return user;
    }
}
```

Controller 层专注业务，无需 try-catch：

```java
@RestController
@RequestMapping("/api/users")
public class UserController {

    @Autowired
    private UserService userService;

    @GetMapping("/{userId}")
    public Result<UserDTO> getUserById(@PathVariable Long userId) {
        UserDTO user = userService.getUserById(userId);
        return Result.success(user);
    }
}
```

### 1.3.7 日志规范

异常日志分级：
- 业务异常（BaseException）：`log.warn`，记录请求路径和错误码
- 系统异常（Exception）：`log.error`，记录完整堆栈

引入 MDC 全链路追踪：在 Filter 或拦截器中为每个请求生成唯一 `traceId` 并存入 MDC，日志配置加入 `%X{traceId}` 字段输出，可通过一个 ID 串联该请求的所有日志，极大提升线上问题排查效率。

## 1.4 传统 Web 项目异常处理方案

传统 Web 项目（服务器端渲染）以 SpringBoot 默认错误处理方案为主，返回 HTML 错误页面。

### 1.4.1 SpringBoot 默认错误处理流程

如果 SpringMVC 没有处理对应异常，`ErrorMvcAutoConfiguration` 会配置 `/error` 端点（白标错误页），按以下顺序查找错误页面：

第一步（精确错误码文件）：
- 先找 `classpath:/templates/error/404.html`、`500.html` 等
- 找不到则去静态资源目录的 `/error/` 下找

第二步（模糊错误码文件）：
- 先找 `classpath:/templates/error/4xx.html`、`5xx.html` 等
- 找不到则去静态资源目录的 `/error/` 下找

第三步（通用错误页面）：
- 找 `classpath:/templates/error.html`

第四步（默认兜底）：
- 以上全部找不到，使用内置 `/error` 端点的默认白标错误响应

### 1.4.2 错误页面中获取错误信息

SpringBoot 默认向 Model 中注入以下字段，在 Thymeleaf 中用 `${字段名}` 取值：

- `timestamp`：错误发生时间戳（默认提供）
- `status`：HTTP 状态码（默认提供）
- `error`：错误类型，如 "Not Found"（默认提供）
- `path`：发生错误的请求路径（默认提供）
- `exception`：异常类名（需配置开启）
- `message`：错误消息（SpringBoot 2.3+ 需配置 `server.error.include-message=always`）
- `trace`：堆栈跟踪（需配置开启）

SpringBoot 3.x 默认只提供 `timestamp`、`status`、`error`、`path` 四项（`exception`、`message`、`trace` 不再默认输出）。`exception`、`message`、`trace` 需显式开启：

```properties
server.error.include-exception=true
server.error.include-message=always
server.error.include-stacktrace=always
```

补充说明：`server.error.include-message` 在 SpringBoot 2.3 之前默认为 `always`，之后改为 `never`，因此低版本升级后如不配置此项，错误消息将不再返回。

### 1.4.3 业务异常处理建议

- HTTP 状态码类错误（404、500 等）：提供精确错误码 HTML 页面，不常见的错误码提供模糊匹配页面
- 业务逻辑异常（非 HTTP 状态码）：在程序中捕获后，手动跳转到对应的错误页面
- 建议始终提供 `classpath:/templates/error.html` 作为兜底通用错误页

## 1.5 HTTP 状态码速查

- 400：参数错误（Bad Request）
- 401：未认证（Unauthorized）
- 403：无权限（Forbidden）
- 404：资源不存在（Not Found）
- 500：系统内部错误（Internal Server Error）

---

## 1.6 进阶方向

- 国际化（i18n）：结合 `MessageSource`，根据请求头 `Accept-Language` 返回本地化错误信息
- 异常分层细化：区分 DaoException、RpcException 等，在全局处理器中做更精细的处理和日志记录
- 异常告警：集成 Sentinel、Prometheus 等监控系统，关键异常触发钉钉 / 短信 / 邮件告警
- 结合 Swagger / OpenAPI：在接口文档中清晰描述各接口的错误码及含义
