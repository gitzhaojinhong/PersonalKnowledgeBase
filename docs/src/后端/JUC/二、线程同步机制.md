## 2.1 synchronized关键字

### 2.1.1 三种使用形式

| 形式 | 锁对象 | 说明 |
|:---|:---|:---|
| **修饰实例方法** | 当前实例（this） | 作用于当前对象实例，锁定该对象的 monitor |
| **修饰静态方法** | 当前类对象（Class 对象） | 锁定整个类，所有实例共享同一把锁 |
| **修饰代码块** | 指定对象（this / xxx.class / 任意对象） | 粒度更细，锁定指定对象 |

```java
public class SynchronizedDemo {

    // 形式1：修饰实例方法 —— 锁住整个实例
    public synchronized void instanceMethod() {
        // 同一实例的所有 synchronized 实例方法互斥
    }

    // 形式2：修饰静态方法 —— 锁住整个类
    public static synchronized void staticMethod() {
        // 所有实例的所有 synchronized 静态方法互斥
    }

    // 形式3：修饰代码块 —— 锁住指定对象
    public void blockMethod() {
        Object lock = new Object();
        synchronized (lock) { // 只锁定这个代码块，粒度更细
            // 临界区代码
        }
    }
}
```

### 2.1.2 底层实现原理

`synchronized` 通过 **monitorenter** 和 **monitorexit** 字节码指令实现：

```java
public void syncMethod() {
    synchronized (this) { // monitorenter
        // 临界区
    } // monitorexit
}
```

每个对象头中都包含一个 **monitor 对象**（也称为管程或监视器锁），当线程执行 monitorenter 时尝试获取对象的 monitor 锁。

### 2.1.3 锁状态与升级

JVM 对 synchronized 进行了锁优化，存在**无锁 → 偏向锁 → 轻量级锁 → 重量级锁**的升级过程。

| 锁状态 | 条件 | 说明 |
|:---|:---|:---|
| **偏向锁** | 单线程反复进入同步块 | 同一线程多次进入只需在对象头记录，无额外开销 |
| **轻量级锁** | 多线程交替进入，无竞争 | 线程通过 CAS 自旋获取锁，不阻塞线程 |
| **重量级锁** | 多线程激烈竞争 | 未获取锁的线程会进入阻塞（BLOCKED）状态 |

> **JDK 21 重要变化**：偏向锁已在 JDK 15 被废弃（JEP 374），在 JDK 18 被完全移除。在 JDK 21 中，synchronized 的锁状态实际只有**无锁 → 轻量级锁 → 重量级锁**三种。下文的四种状态描述适用于 JDK 8~14，供理解锁演进历史。

```java
// 企业级使用场景示例：库存扣减（并发安全）
public class InventoryService {
    private int stock = 1000;

    public synchronized boolean deductStock() {
        if (stock > 0) {
            stock--;
            return true;
        }
        return false;
    }
}
```

> **注意**：JDK 1.6 之后 synchronized 性能已大幅提升，大多数场景可直接使用，无需过早优化。

### 2.1.4 锁升级详细流程

锁的状态存储在对象头的 **Mark Word** 中，64 位 JVM 的 Mark Word 结构如下：

| 锁状态 | 存储内容 |
|:---|:---|
| **无锁** | 对象哈希码 + 分代年龄 + 0（偏向锁位）+ 01（锁标志位） |
| **偏向锁** | 线程 ID + 分代年龄 + 1（偏向锁位）+ 01 |
| **轻量级锁** | 指向栈中锁记录的指针 + 00 |
| **重量级锁** | 指向互斥量（monitor）的指针 + 10 |

**升级流程（JDK 8~14，含偏向锁）**：

```
无锁状态
    │
    ▼  第一个线程获取锁
偏向锁（记录线程ID）
    │
    ├── 同一线程再次进入 → 无需任何同步，直接访问（最快）
    │
    ▼  第二个线程尝试获取（竞争）
轻量级锁（CAS自旋）
    │
    ├── 自旋成功 → 获取锁（无阻塞）
    │
    ▼  自旋超过阈值（默认10次）或竞争加剧
重量级锁（操作系统Mutex，线程阻塞）
```

**JDK 21 实际流程（无偏向锁）**：

```
无锁状态
    │
    ▼  第一个线程获取锁
轻量级锁（CAS自旋）
    │
    ├── 自旋成功 → 获取锁（无阻塞）
    │
    ▼  自旋超过阈值或竞争加剧
重量级锁（操作系统Mutex，线程阻塞）
```

**各状态转换条件**：

| 转换 | 触发条件 |
|:---|:---|
| 无锁 → 偏向锁 | 第一个线程进入同步块，JVM 启用偏向锁（默认开启） |
| 偏向锁 → 轻量级锁 | 其他线程尝试获取锁，发生竞争 |
| 轻量级锁 → 重量级锁 | 自旋次数超过阈值（默认 10 次）或有更多线程竞争 |
| 重量级锁 → 轻量级锁 | 一般不会发生（锁升级是单向的） |

> **JDK 15 及以后**：偏向锁已被**废弃**（JEP 374），因为现代应用中多线程竞争频繁，偏向锁的收益小于维护成本。启动参数 `-XX:+UseBiasedLocking` 从 JDK 18 起不再生效。

## 2.2 volatile关键字

### 2.2.1 核心语义

`volatile` 是 Java 最轻量级的同步机制，保证两个特性：

- **可见性**：对 volatile 变量的写立即刷新到主内存，读直接从主内存读取
- **有序性**：禁止指令重排序（插入内存屏障）

```java
public class VolatileDemo {
    // volatile 保证多线程下 one = true 的写操作对其他线程可见
    private volatile boolean one = false;

    public void writer() {
        one = true; // 写操作 —— 强制刷入主内存
    }

    public void reader() {
        if (one) { // 读操作 —— 强制从主内存读取
            // 一定能读到 one=true
        }
    }
}
```

### 2.2.2 不保证原子性

```java
public class VolatileNotAtomic {
    private volatile int count = 0;

    // 此处即使 volatile 也不安全！
    // i++ 是"读取-修改-写入"三步操作，非原子
    public void increment() {
        count++; // 非原子操作，volatile 无法保证
    }

    // 正确做法：使用原子类
    private AtomicInteger atomicCount = new AtomicInteger(0);

    public void safeIncrement() {
        atomicCount.incrementAndGet(); // 原子操作
    }
}
```

### 2.2.3 应用场景

| 场景 | 示例 |
|:---|:---|
| **状态标志** | `volatile boolean running = true;`（单例的双重检查） |
| **双重检查锁定** | 单例模式中 `private static volatile Singleton instance;` |
| **屏障前后操作** | 保证读操作能看到写操作之前的全部结果 |

```java
// 场景一：状态标志
public class Worker extends Thread {
    private volatile boolean running = true;

    @Override
    public void run() {
        while (running) {
            // 处理任务
        }
    }

    public void stopThread() {
        running = false; // 修改对所有线程立即可见
    }
}

// 场景二：单例双重检查锁定（DCL）
public class Singleton {
    // volatile 禁止重排序，防止指令重排导致对象未完全初始化就被其他线程引用
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) { // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) { // 第二次检查
                    instance = new Singleton(); // volatile 防止指令重排
                }
            }
        }
        return instance;
    }
}
```

### 2.2.4 内存屏障机制

`volatile` 通过插入**内存屏障（Memory Barrier / Fence）** 实现可见性和有序性。JVM 在不同 CPU 架构下会插入不同类型的屏障指令：

| 屏障类型 | 作用 | 插入位置 |
|:---|:---|:---|
| **LoadLoad** | 禁止读-读重排序 | volatile 读操作后 |
| **LoadStore** | 禁止读-写重排序 | volatile 读操作后 |
| **StoreStore** | 禁止写-写重排序 | volatile 写操作前 |
| **StoreLoad** | 禁止写-读重排序（最"重"的屏障） | volatile 写操作后 |

```
普通写操作;  StoreStore屏障;  volatile写;  StoreLoad屏障;
                                                            volatile读;  LoadLoad屏障;  LoadStore屏障;  普通读/写
```

**关键理解**：`StoreLoad` 是最昂贵的屏障，它会刷新处理器缓存到主内存，并使其他 CPU 的缓存失效。因此 `volatile` 写操作的开销大于读操作。

## 2.3 wait与notify机制

### 2.3.1 基本用法

`Object.wait()`、`notify()` 和 `notifyAll()` 是 JDK 早期提供的线程协作机制，必须在 **synchronized 代码块内**调用。

| 方法 | 说明 |
|:---|:---|
| `wait()` | 当前线程进入对象的等待队列，**释放已持有的锁** |
| `wait(long timeout)` | 带超时的等待，超时后自动唤醒 |
| `notify()` | 唤醒等待队列中的一个线程（由 JVM 决定哪个） |
| `notifyAll()` | 唤醒等待队列中的所有线程 |

```java
public class WaitNotifyDemo {
    private final Object lock = new Object();
    private boolean condition = false;

    // 等待方
    public void await() throws InterruptedException {
        synchronized (lock) {
            while (!condition) { // 必须用 while，避免虚假唤醒
                lock.wait(); // 释放锁并等待
            }
            // condition == true 时继续执行
        }
    }

    // 通知方
    public void signal() {
        synchronized (lock) {
            condition = true;
            lock.notifyAll(); // 唤醒所有等待线程
        }
    }
}
```

> **重要**：等待必须使用 while (!condition) 循环而非 if，因为存在虚假唤醒（spurious wakeup）——线程可能在没有被显式 notify 的情况下被唤醒，必须重新检查条件是否真正满足。

### 2.3.2 生产者-消费者模式

```java
public class ProducerConsumer {
    private final int MAX_SIZE = 10;
    private final Queue<Integer> queue = new LinkedList<>();

    // 生产者
    public void produce(int value) throws InterruptedException {
        synchronized (queue) {
            while (queue.size() >= MAX_SIZE) {
                queue.wait(); // 队列满，等待消费
            }
            queue.add(value);
            queue.notifyAll(); // 唤醒等待消费的线程
        }
    }

    // 消费者
    public Integer consume() throws InterruptedException {
        synchronized (queue) {
            while (queue.isEmpty()) {
                queue.wait(); // 队列空，等待生产
            }
            Integer value = queue.poll();
            queue.notifyAll(); // 唤醒等待生产的线程
            return value;
        }
    }
}
```

## 2.4 线程安全策略

除了使用锁和同步机制，还可以通过**设计层面的策略**来实现线程安全，避免锁带来的性能开销。

### 2.4.1 不可变对象

不可变对象一旦创建，其状态就不能被修改，因此天然线程安全。

```java
// ✅ 不可变对象示例
public final class ImmutableUser {
    private final String name;      // final 保证引用不可变
    private final int age;
    private final List<String> tags; // 引用不可变，但内部内容可能可变

    public ImmutableUser(String name, int age, List<String> tags) {
        this.name = name;
        this.age = age;
        // 防御性拷贝，确保内部状态不可被外部修改
        this.tags = new ArrayList<>(tags);
    }

    // 只有 getter，没有 setter
    public String getName() { return name; }
    public int getAge() { return age; }
    public List<String> getTags() {
        // 返回不可修改视图或拷贝
        return Collections.unmodifiableList(tags);
    }
}
```

Java 中常见的不可变对象：`String`、`Integer`、`Long` 等包装类、`java.time` 包中的日期时间类。

### 2.4.2 线程封闭

将数据限制在单个线程内访问，其他线程无法访问，从而避免同步。

| 实现方式 | 说明 |
|:---|:---|
| **栈封闭** | 局部变量存储在栈中，线程私有，天然线程安全 |
| **ThreadLocal** | 每个线程拥有独立副本（详见第九章） |

```java
public class StackConfinement {
    // 局部变量是线程安全的（栈封闭）
    public void safeMethod() {
        int count = 0; // 局部变量，线程私有
        count++;
        System.out.println(count);
    }
}
```

### 2.4.3 同步容器

JDK 早期提供的通过 `synchronized` 包装的容器，所有方法都加锁，并发度低。

```java
// 同步容器（通过 Collections 工具类包装）
List<String> syncList = Collections.synchronizedList(new ArrayList<>());
Map<String, String> syncMap = Collections.synchronizedMap(new HashMap<>());
Set<String> syncSet = Collections.synchronizedSet(new HashSet<>());
```

> **注意**：同步容器的**迭代器不是线程安全**的，需要手动加锁。且所有方法都串行执行，高并发下性能差。生产环境优先使用 JUC 并发集合。

| 容器类型 | 代表类 | 并发策略 | 适用场景 |
|:---|:---|:---|:---|
| **同步容器** | `Vector`、`Hashtable`、`Collections.synchronizedXXX` | 方法级 `synchronized` | 低并发，简单场景 |
| **并发容器** | `ConcurrentHashMap`、`CopyOnWriteArrayList` | 分段锁 / CAS / 写时复制 | 高并发场景 |
