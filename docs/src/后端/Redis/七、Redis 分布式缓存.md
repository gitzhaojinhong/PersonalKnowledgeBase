
## 7.1 数据持久化

Redis 的数据存储在内存中，需要持久化到磁盘以防止数据丢失。主要有 RDB 和 AOF 两种方式。

### 7.1.1 RDB（Redis DataBase）

RDB 是 **数据快照** 方式。将某个时刻 Redis 内存中的所有数据记录到dump.rdb文件中。

**触发方式**：

**方式一：手动触发**
```bash
# 阻塞方式（阻塞主线程，不推荐生产使用）
redis-cli bgsave

# 非阻塞方式（fork 子进程处理）
redis-cli save
```

**方式二：自动触发**

在 redis.conf 中配置：
```properties
# 900秒内至少1个key变化时触发
save 900 1
# 300秒内至少10个key变化时触发
save 300 10
# 60秒内至少10000个key变化时触发
save 60 10000
```

**RDB 优缺点**：
- 优点：文件紧凑、全量备份、恢复速度快（直接加载rdb文件）
- 缺点：最后一次快照之后的数据会丢失；fork 子进程时如果数据量大，阻塞时间可能较长

### 7.1.2 AOF（Append Only File）

AOF 是 **命令日志** 方式。每执行一条写命令，就追加到 aof 文件末尾。

**配置**：

```properties
# 开启AOF
appendonly yes

# 同步策略（always/everysec/no）
# everysec：每秒同步一次，最多丢1秒数据（推荐）
appendfsync everysec

# AOF文件大于64MB时自动重写
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

**AOF 重写**：随着命令越来越多，AOF 文件会越来越大。重写会fork子进程，读取当前Redis数据，生成最小命令集写入新文件，然后替换旧文件。

**AOF 三种同步策略**：

| 策略 | 安全性 | 性能 |
|---|---|---|
| `always` | 每条命令都同步，几乎不丢数据 | 最低 |
| `everysec`（默认） | 每秒同步，最多丢1秒数据 | 适中 |
| `no` | 由操作系统决定何时同步 | 最高 |

### 7.1.3 混合持久化

Redis 4.0 引入了混合持久化，rdb文件开头 + aof增量数据：

```properties
aof-use-rdb-preamble yes
```

恢复时先加载 rdb 部分提速，再追加后续的 aof 日志。生产推荐使用 **RDB + AOF 混合模式**。

## 7.2 主从复制

### 7.2.1 概念

主从复制是指将主节点（Master）的数据实时复制到从节点（Slave）。主节点负责写，从节点负责读，实现读写分离。

**作用**：
- 数据冗余（备份）
- 读写分离（分担负载）
- 故障恢复（从节点顶替主节点）

### 7.2.2 配置主从

**方式一：命令配置（临时生效）**

```bash
# 在从节点上执行
redis-cli replicaof <主节点IP> <主节点端口>

# 取消从属关系（恢复为主节点）
redis-cli replicaof no one
```

**方式二：配置文件（永久生效）**

在从节点的 redis.conf 中添加：
```properties
replicaof <主节点IP> <主节点端口>
masterauth <主节点密码>   # 如果主节点有密码
replica-read-only yes     # 从节点只读
```

### 7.2.3 复制原理

**全量同步**（初次同步）：
1. 从节点请求主节点同步数据
2. 主节点执行 bgsave 生成 RDB 快照
3. 主节点将 RDB 发送给从节点
4. 从节点清空本地数据，加载 RDB
5. 主节点将快照期间的新命令（replication buffer）发送给从节点
6. 从节点执行这些命令，完成全量同步

**增量同步**（同步完成后）：
主节点每执行一条写命令，就将命令发送给从节点（通过 replication stream）。

**replication backlog**：主节点维护一个环形缓冲区，记录最近执行的写命令。从节点重连后，如果差距不大（还在缓冲区范围内），就做增量同步；否则做全量同步。

## 7.3 哨兵机制

### 7.3.1 概念

主从复制模式下，如果主节点故障，需要人工手动干预（执行 `replicaof no one`）切换从节点，不够自动化。**哨兵机制（Sentinel）** 就是来解决这个问题的。

**哨兵功能**：
- **监控**：持续监控主从节点健康状态
- **自动故障转移**：主节点故障时，自动从从节点中选举新主节点
- **通知**：故障转移后通知应用端新地址（通过发布订阅）
- **选主规则**：哨兵根据从节点的优先级、复制进度、节点ID等综合评估

### 7.3.2 配置哨兵

创建 sentinel.conf 配置文件：

```properties
# 端口
port 26379

# 监控的主节点：名称  IP  端口  投票阈值(几票认定故障)
sentinel monitor mymaster 192.168.150.101 6379 2

# 密码（如果主节点有密码）
sentinel auth-pass mymaster 123321

# 主观下线时间（ms）
sentinel down-after-milliseconds mymaster 10000

# 故障转移超时时间
sentinel failover-timeout mymaster 180000

# 同时向多少个从节点发起同步
sentinel parallel-syncs mymaster 1
```

启动哨兵：
```bash
redis-sentinel sentinel.conf
```

### 7.3.3 核心原理

**主观下线（SDown）**：单个哨兵认为主节点不可用。
**客观下线（ODown）**：多个哨兵投票，超过阈值认为主节点真的故障了。

**故障转移流程**：
1. 多个哨兵认为主节点 SDown → ODown
2. 哨兵集群选举出一个 leader 负责执行故障转移
3. leader 从从节点中选出一个作为新主节点
4. leader 向其他从节点发送 replicaof 命令指向新主节点
5. leader 更新配置，向外宣告新主节点地址

## 7.4 分片集群

### 7.4.1 概念

主从复制 + 哨兵解决了读写分离和故障自动切换，但数据量受单机内存限制。**分片集群（Cluster）** 通过数据分片，每个节点只存储部分数据，横向扩展存储容量。

**架构**：
- 16384 个槽（slot），分布在多个节点上
- 每个节点存储一段范围的槽（可以配置主从）
- 客户端访问任意节点，数据自动路由到正确节点

### 7.4.2 搭建分片集群

**拓扑**：3主3从（每台机器一个主节点一个从节点）

| 节点 | 端口 | 角色 |
|---|---|---|
| 192.168.150.101 | 7001 | 主 |
| 192.168.150.101 | 7002 | 从（7001的副本） |
| 192.168.150.102 | 7001 | 主 |
| 192.168.150.102 | 7002 | 从（7001的副本） |
| 192.168.150.103 | 7001 | 主 |
| 192.168.150.103 | 7002 | 从（7001的副本） |

**每个节点的 redis.conf 配置**：

```properties
port 7001
daemonize yes
cluster-enabled yes
cluster-config-file nodes-7001.conf
cluster-node-timeout 5000
```

**创建集群**（使用 redis-cli）：

```bash
redis-cli --cluster create \
  192.168.150.101:7001 \
  192.168.150.101:7002 \
  192.168.150.102:7001 \
  192.168.150.102:7002 \
  192.168.150.103:7001 \
  192.168.150.103:7002 \
  --cluster-replicas 1
```

**分配槽**：`--cluster-replicas 1` 表示每个主节点配1个从节点，集群会自动分配槽。

### 7.4.3 集群伸缩

**添加新节点**：

```bash
# 添加新主节点
redis-cli --cluster add-node 192.168.150.103:7003 192.168.150.101:7001

# 给新主节点分配槽
redis-cli --cluster reshard 192.168.150.101:7001 \
  --cluster-to <新节点ID> \
  --cluster-slots <槽数量> \
  --cluster-from <源节点ID1>,<源节点ID2>
```

### 7.4.4 故障转移

集群中的从节点如果检测到主节点不可用，会自动发起选举，争夺成为新主节点。选举逻辑与哨兵类似（基于投票数）。

**注意**：Redis Cluster 要求至少3个主节点才能正常工作，每个主节点至少1个从节点才能保证高可用。
