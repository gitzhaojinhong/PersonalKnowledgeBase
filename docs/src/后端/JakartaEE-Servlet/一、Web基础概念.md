
## 1.1 软件架构模式

### C/S架构（Client/Server）

客户端-服务器架构，需要在用户设备上安装专门的客户端程序。

**特点：**
- 需要安装专用客户端软件
- 客户端可充分利用本地硬件资源
- 部分业务逻辑在客户端处理
- 升级时需要更新所有客户端

**典型应用：** QQ、微信PC版、游戏客户端、银行U盾系统

### B/S架构（Browser/Server）

浏览器-服务器架构，用户通过浏览器访问应用，无需安装客户端。

**特点：**
- 无需安装客户端，浏览器即可访问
- 业务逻辑集中在服务器端
- 升级只需更新服务器
- 跨平台性好，只要有浏览器即可

**典型应用：** 电商网站、OA系统、在线文档、各类管理后台

### 架构对比

| 特性 | C/S架构 | B/S架构 |
|------|---------|---------|
| 安装要求 | 需安装专用客户端 | 只需浏览器 |
| 维护成本 | 高（需更新所有客户端） | 低（仅更新服务器） |
| 性能 | 高（可利用本地资源） | 依赖网络和服务端 |
| 跨平台性 | 差 | 好 |
| 安全性 | 可本地加密，相对安全 | 需考虑传输安全 |
| 适用场景 | 复杂业务、高性能需求 | 信息发布、管理类系统 |

## 1.2 Web资源分类

### 静态资源

内容固定，服务器直接返回，无需处理。

**常见类型：**
- HTML页面
- CSS样式文件
- JavaScript脚本
- 图片（JPG、PNG、GIF等）
- 字体文件、视频、音频

**特点：**
- 所有用户看到的内容相同
- 服务器只做简单读取和传输
- 可通过CDN加速分发

### 动态资源

内容由服务器根据请求动态生成，通常涉及数据库查询和业务处理。

**常见类型：**
- Servlet生成的页面
- JSP/Thymeleaf模板渲染结果
- REST API接口返回的JSON数据

**特点：**
- 内容因人而异、因时而异
- 需要服务器执行业务逻辑
- 通常涉及数据库操作

## 1.3 系统间通信方式

### 方式一：基于请求响应（HTTP协议）

**适用场景：** 实时性要求不高的场景，如网页浏览、API调用。

**特点：**
- 客户端主动发起请求
- 服务器被动响应
- 无请求则无响应
- 基于HTTP协议，简单通用

```
浏览器 → HTTP请求 → Web服务器 → 处理 → HTTP响应 → 浏览器
```

### 方式二：直接访问对方数据库

**适用场景：** 内部系统间数据共享，如报表系统直接读取业务库。

**特点：**
- 实时性高
- 耦合度高，安全性差
- 可能暴露敏感数据
- 不推荐跨组织使用

**缺点：**
- 破坏封装性，直接暴露数据结构
- 安全性难以控制
- 一方数据库变更影响另一方

### 方式三：基于消息队列（MQ）

**适用场景：** 高并发、异步处理、系统解耦，如订单系统与库存系统通信。

**特点：**
- 异步通信，发送方无需等待
- 系统间解耦，互不影响
- 支持削峰填谷，应对流量高峰
- 保证消息可靠传递

```
系统A → 发送消息 → MQ → 系统B消费消息
```

**常见MQ产品：** RabbitMQ、Kafka、RocketMQ

## 1.4 URL结构详解

URL（统一资源定位符）是互联网上资源的完整地址，格式如下：

```
  https://www.example.com:8080/path/to/page
  \___/   \_____________/ \__/ \_________/
    |           |          |        |
  协议      主机名（域名）  端口     路径
```

**各部分说明：**

- **协议**：`http` 表示普通网页协议，`https` 表示加密安全协议（SSL/TLS）。HTTP协议分为请求协议（B→S）和响应协议（S→B）两部分。
- **主机名**：可以是 IP 地址（如 `9.23.56.78`）或域名（如 `www.example.com`）。域名通过 DNS 解析到 IP 地址。
- **端口**：定位服务器上的某个服务。Web 服务默认端口 80（HTTPS 默认 443），Tomcat 默认 8080。
- **路径**：Web 服务器上某个资源的路径，如 `/oa/index.html` 表示 oa 项目下的 index.html。

## 1.5 BS通信完整流程

以用户访问 `https://www.example.com` 为例：

```
用户输入URL
    ↓
DNS解析（本地缓存 → 本地DNS → 权威DNS → 得到IP）
    ↓
TCP三次握手（建立连接）
    ↓
TLS握手（HTTPS专有，交换证书、协商加密密钥）
    ↓
浏览器发送HTTP请求报文（GET /index.html HTTP/1.1）
    ↓
服务器处理（Tomcat → Servlet → 业务逻辑 → 数据库）
    ↓
服务器返回HTTP响应报文（200 OK + HTML内容）
    ↓
浏览器解析HTML，构建DOM树，渲染页面
    ↓
断开连接（HTTP/1.0默认短连接，HTTP/1.1默认长连接复用）
```

## 1.6 URL和URI的区别

- **URI（统一资源标识符）**：唯一标识一个资源，范围更大，包含 URL 和 URN。
- **URL（统一资源定位符）**：不仅标识资源，还提供访问方式（协议 + 地址），是 URI 的子集。

类比理解：URI 像"身份证号"——唯一标识一个人，但不告诉你如何找到他；URL 像"家庭住址"——不仅标识这个人，还告诉你如何去他家。

## 1.7 BS系统涉及的角色与协议

一个完整的 BS 系统涉及 **4 个角色** 和 **3 个协议**：

**4 个角色：**
- 浏览器开发者（开发 Chrome、Firefox 的工程师）
- Web 服务器开发者（开发 Tomcat 的工程师）
- Web 应用开发者（JavaWeb 程序员，就是我们）
- 数据库服务器开发者（开发 MySQL 的工程师）

**3 个协议：**
- **HTTP**：浏览器与 Web 服务器之间的通信协议
- **Servlet 规范**：Web 服务器与 Web 应用之间的规范。遵守 Servlet 规范，Web 应用才能部署到任何符合规范的容器中（解耦合）
- **JDBC 规范**：Java 程序与数据库之间的规范。遵守 JDBC 规范，不改代码即可切换 MySQL、Oracle 等数据库（解耦合）
