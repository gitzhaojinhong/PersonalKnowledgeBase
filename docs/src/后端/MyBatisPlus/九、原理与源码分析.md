## 九、原理与源码分析

### 9.1 SQL 自动注入原理

#### 9.1.1 核心流程

MyBatis-Plus 启动时自动扫描继承了 `BaseMapper` 的接口，通过 **SQL 注入器** 将通用 CRUD 方法的 SQL 注入到 MyBatis 的 `Configuration` 中。

```
1. Spring 启动
   └─> MapperScannerConfigurer 扫描 @Mapper 接口
       └─> 发现 UserMapper extends BaseMapper<User>
           └─> MybatisMapperProxyFactory 创建代理
               └─> DefaultSqlInjector.inject()
                   └─> 遍历内置方法列表（Insert、DeleteById、UpdateById、SelectById...）
                       └─> 每个方法生成对应的 MappedStatement
                           └─> 注入到 Mybatis Configuration.mappedStatements
```

#### 9.1.2 SQL 模板解析

以 `selectById` 为例，MyBatis-Plus 使用预定义的 SQL 模板：

```java
// SelectById 注入方法
public class SelectById extends AbstractMethod {
    
    @Override
    public MappedStatement injectMappedStatement(Class<?> mapperClass, Class<?> modelClass,
                                                   TableInfo tableInfo) {
        // SQL 模板：SELECT {columns} FROM {table} WHERE {pk} = #{id}
        SqlMethod sqlMethod = SqlMethod.SELECT_BY_ID;
        
        SqlSource sqlSource = new RawSqlSource(configuration, 
            String.format(sqlMethod.getSql(), 
                sqlSelectColumns(tableInfo, false),
                tableInfo.getTableName(), 
                tableInfo.getKeyColumn(), 
                tableInfo.getKeyProperty(),
                tableInfo.getLogicDeleteSql(true, true)),
            Object.class);
        
        return this.addSelectMappedStatementForTable(mapperClass, 
            sqlMethod.getMethod(), sqlSource, tableInfo);
    }
}
```

**生成的 SQL 示例**：

```sql
-- BaseMapper.selectById(1L)
SELECT id, username, age, email, create_time, update_time, deleted 
FROM t_user 
WHERE id = 1 AND deleted = 0
```

### 9.2 插件机制（Interceptor）

MyBatis-Plus 3.4.0+ 采用 **责任链模式** 的插件机制，所有插件统一通过 `MybatisPlusInterceptor` 管理。

#### 9.2.1 插件结构

```java
public class MybatisPlusInterceptor implements Interceptor {
    
    private List<InnerInterceptor> interceptors = new ArrayList<>();
    
    public void addInnerInterceptor(InnerInterceptor interceptor) {
        this.interceptors.add(interceptor);
    }
    
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        // 遍历所有内层拦截器，依次执行
        // 分页拦截器 -> 多租户拦截器 -> 乐观锁拦截器 -> ...
    }
}
```

#### 9.2.2 内置 InnerInterceptor

| 拦截器 | 功能 |
|--------|------|
| `PaginationInnerInterceptor` | 分页查询拦截 |
| `TenantLineInnerInterceptor` | 多租户 SQL 改写 |
| `DynamicTableNameInnerInterceptor` | 动态表名替换 |
| `OptimisticLockerInnerInterceptor` | 乐观锁版本号处理 |
| `BlockAttackInnerInterceptor` | 防止全表更新/删除 |

#### 9.2.3 防止全表更新/删除（BlockAttackInnerInterceptor）

```java
@Bean
public MybatisPlusInterceptor mybatisPlusInterceptor() {
    MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
    
    // 添加防全表更新/删除插件
    interceptor.addInnerInterceptor(new BlockAttackInnerInterceptor());
    
    return interceptor;
}
```

配置后，不带 WHERE 条件的 UPDATE/DELETE 将被拦截并抛出异常：

```java
// 以下代码将抛出异常：MyBatisPlusException: Prohibit full table update/delete
userMapper.update(null, null);
userMapper.delete(null);
```

#### 9.2.4 非法 SQL 拦截（IllegalSQLInnerInterceptor）

`IllegalSQLInnerInterceptor` 用于拦截无意义或低性能的 SQL（如未带 WHERE 条件的 SELECT、UPDATE、DELETE）：

```java
@Bean
public MybatisPlusInterceptor mybatisPlusInterceptor() {
    MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
    
    // 添加非法 SQL 拦截插件
    interceptor.addInnerInterceptor(new IllegalSQLInnerInterceptor());
    
    return interceptor;
}
```

**与 BlockAttackInnerInterceptor 的区别**：

| 插件 | 拦截场景 |
|------|---------|
| `BlockAttackInnerInterceptor` | 无 WHERE 条件的 UPDATE / DELETE |
| `IllegalSQLInnerInterceptor` | 无 WHERE 条件的 SELECT / UPDATE / DELETE，以及明显低性能的 SQL |

### 9.3 条件构造器原理

#### 9.3.1 Wrapper 结构

```java
public abstract class AbstractWrapper<T, R, Children extends AbstractWrapper<T, R, Children>>
    extends Wrapper<T> {
    
    // SQL 片段集合
    protected final MergeSegments expression;
    
    // 参数名-值映射
    protected final Map<String, Object> paramNameValuePairs;
    
    // 构建 WHERE 条件
    public Children eq(R column, Object val) {
        return addCondition(true, column, SqlKeyword.EQ, val);
    }
}
```

#### 9.3.2 SQL 片段生成

当调用 `userMapper.selectList(wrapper)` 时：

```
1. MybatisMapperProxy 拦截方法调用
   └─> SqlSession.selectList("com.example.mapper.UserMapper.selectList", wrapper)
       └─> MP 解析 wrapper
           └─> wrapper.getCustomSqlSegment() 生成 "WHERE age > 18 AND username LIKE '%张%'"
               └─> 替换到预定义 SQL 模板中
                   └─> 最终 SQL：SELECT * FROM t_user WHERE age > 18 AND username LIKE '%张%' AND deleted = 0
```

### 9.4 分页插件原理

#### 9.4.1 物理分页实现

MyBatis-Plus 的分页插件通过拦截 `Executor.query` 方法，对 SQL 进行两次处理：

1. **COUNT 查询**：先执行 `SELECT COUNT(*) FROM ...` 获取总记录数
2. **分页查询**：将原始 SQL 改写为带 `LIMIT`（MySQL）或 `ROWNUM`（Oracle）的分页 SQL

```java
// 原始 SQL
SELECT * FROM t_user WHERE age > 18

// 改写后（MySQL）
SELECT * FROM t_user WHERE age > 18 LIMIT 0, 10

// COUNT SQL
SELECT COUNT(*) FROM t_user WHERE age > 18
```

#### 9.4.2 分页优化

```java
// 当确定知道总记录数时，可跳过 COUNT 查询
Page<User> page = new Page<>(1, 10, false);  // 第三个参数 false 表示不执行 COUNT

// 或者传入已知的总记录数
Page<User> page = new Page<>(1, 10, 1000L);  // 已知共 1000 条
```
