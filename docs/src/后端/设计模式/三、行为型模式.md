# 三、行为型模式

## 3.1 策略设计模式

### 3.1.1 模式定义 / 概述
策略模式（Strategy Pattern）定义一系列算法，将每个算法封装起来，使它们可以互相替换。算法的变化不会影响使用算法的客户端。属于**行为型**设计模式。

核心思想：将算法族提取为接口，客户端面向接口编程，运行时注入具体策略，消除 if-else / switch-case。

### 3.1.2 核心角色 / 结构组成
- **Strategy（策略接口）**：定义算法的公共接口。
- **ConcreteStrategy（具体策略）**：实现具体算法。
- **Context（上下文）**：持有策略对象引用，委托策略执行算法。

### 3.1.3 工作流程 / 类图
```
Context ──持有──▶ Strategy(接口)
                     ├── ConcreteStrategyA
                     ├── ConcreteStrategyB
                     └── ConcreteStrategyC
```

### 3.1.4 代码示例
```java
// 策略接口：排序算法
public interface SortStrategy {
    void sort(int[] arr);
}

// 具体策略A：冒泡排序
public class BubbleSortStrategy implements SortStrategy {
    @Override
    public void sort(int[] arr) {
        System.out.println("使用冒泡排序");
        for (int i = 0; i < arr.length - 1; i++)
            for (int j = 0; j < arr.length - 1 - i; j++)
                if (arr[j] > arr[j+1]) { int t = arr[j]; arr[j] = arr[j+1]; arr[j+1] = t; }
    }
}

// 具体策略B：快速排序（简化）
public class QuickSortStrategy implements SortStrategy {
    @Override
    public void sort(int[] arr) {
        System.out.println("使用快速排序");
        Arrays.sort(arr); // 简化示意
    }
}

// 上下文
public class Sorter {
    private SortStrategy strategy;
    public Sorter(SortStrategy strategy) { this.strategy = strategy; }
    public void setStrategy(SortStrategy strategy) { this.strategy = strategy; }
    public void sort(int[] arr) { strategy.sort(arr); }
}

// 客户端：运行时切换策略，无需修改 Sorter
public class Client {
    public static void main(String[] args) {
        int[] data = {5, 3, 1, 4, 2};
        Sorter sorter = new Sorter(new BubbleSortStrategy());
        sorter.sort(data); // 使用冒泡排序

        sorter.setStrategy(new QuickSortStrategy());
        sorter.sort(data); // 使用快速排序
    }
}
```

### 3.1.5 优缺点
**优点：**
- 消除大量 if-else，代码更清晰。
- 符合OCP，新增策略无需修改上下文。
- 策略可以自由切换和复用。

**缺点：**
- 策略类数量增多。
- 客户端需要了解所有策略才能正确选择。

### 3.1.6 应用场景
- 多种算法可以互换时（排序、压缩、加密）。
- 需要消除 if-else 多分支逻辑时。

### 3.1.7 经典框架实现
**Spring 的 Resource 资源加载策略：**
```java

```


## 3.2 模板方法设计模式

### 3.2.1 模式定义 / 概述
模板方法模式（Template Method Pattern）在父类中定义一个算法的骨架，将某些步骤的具体实现延迟到子类。属于**行为型**设计模式。

核心思想：不变的部分（算法骨架）放父类，可变的部分（具体实现）留给子类，通过钩子方法让子类控制某些环节的执行与否。

### 3.2.2 核心角色 / 结构组成
- **AbstractClass（抽象类）**：定义模板方法（`final`）和若干抽象/钩子方法。
- **ConcreteClass（具体子类）**：实现抽象方法，按需重写钩子方法。

### 3.2.3 工作流程 / 类图
```
AbstractClass
  ├── templateMethod()（final，骨架）
  ├── step1()（abstract，子类必须实现）
  ├── step2()（abstract，子类必须实现）
  └── hook()（钩子，有默认实现，子类可覆盖）
```

### 3.2.4 代码示例
```java
// 抽象类：游戏流程模板
abstract class Game {
    // 模板方法（final，禁止子类修改骨架）
    public final void play() {
        start();
        playing();
        if (needSound()) { playSound(); }
    }

    protected abstract void start();    // 抽象步骤：必须实现
    protected abstract void playing();  // 抽象步骤：必须实现

    private void playSound() { System.out.println("播放游戏音效..."); }

    // 钩子方法：子类可决定是否开启音效（默认开启）
    protected boolean needSound() { return true; }
}

// 具体子类：象棋
class ChessGame extends Game {
    @Override
    protected void start() { System.out.println("摆好棋盘，红方先走"); }
    @Override
    protected void playing() { System.out.println("车马炮相士帅，轮流走子"); }
    @Override
    protected boolean needSound() { return false; } // 象棋不需要音效
}

// 具体子类：赛车
class RacingGame extends Game {
    @Override
    protected void start() { System.out.println("3、2、1，踩油门出发！"); }
    @Override
    protected void playing() { System.out.println("控制方向盘，躲避障碍物"); }
    // 使用默认 needSound()，赛车需要音效
}

public class Test {
    public static void main(String[] args) {
        new ChessGame().play();
        System.out.println("---");
        new RacingGame().play();
    }
}
```

> **注意：** 模板方法不一定要有钩子，但钩子是最佳实践——让子类能有限度地影响算法流程。

### 3.2.5 优缺点
**优点：**
- 公共逻辑集中在父类，代码复用高。
- 子类实现扩展，而非修改，符合OCP。

**缺点：**
- 每个不同的实现都要子类化，子类数量可能增多。
- 子类执行结果影响父类，增加了理解难度。

### 3.2.6 应用场景
- 多个子类有相同的算法骨架，但具体步骤不同时。
- 控制子类扩展，只允许在特定点插入逻辑时。

### 3.2.7 经典框架实现 / 源码剖析
**HttpServlet 源码（模板方法模式）：**

`HttpServlet` 中的 `service(HttpServletRequest, HttpServletResponse)` 是模板方法，根据请求方式分发到 `doGet()`、`doPost()` 等钩子方法：

```java
// HttpServlet 简化源码
protected void service(HttpServletRequest req, HttpServletResponse resp) {
    String method = req.getMethod();
    if (method.equals("GET")) {
        doGet(req, resp);      // 钩子方法，子类重写
    } else if (method.equals("POST")) {
        doPost(req, resp);     // 钩子方法，子类重写
    }
    // ...其他方法
}
```

开发者继承 `HttpServlet` 后只需重写 `doGet()`/`doPost()`，无需关心请求分发逻辑。

**Spring 的 JdbcTemplate（同时体现了外观模式，见§11外观模式）：**

`JdbcTemplate` 的 `execute()` 是模板方法，负责连接获取、异常处理、连接释放等固定流程，将具体 SQL 执行委托给 `StatementCallback` 回调（策略/钩子）：
```java
// 使用 JdbcTemplate（模板方法 + 回调）
jdbcTemplate.query("SELECT * FROM user WHERE id = ?",
    (rs, rowNum) -> new User(rs.getLong("id"), rs.getString("name")), // 钩子：行映射
    userId
);
```

## 3.3 责任链设计模式

### 3.3.1 模式定义 / 概述
责任链模式（Chain of Responsibility Pattern）将请求的发送者和接收者解耦，让多个对象都有机会处理同一请求，将这些对象连成一条链，沿链传递请求直到被处理。属于**行为型**设计模式。

### 3.3.2 核心角色 / 结构组成
- **Handler（抽象处理者）**：定义处理接口，持有下一个处理者的引用。
- **ConcreteHandler（具体处理者）**：判断能否处理，能则处理，否则转发。
- **Client（客户端）**：组装责任链，向链头发起请求。

### 3.3.3 工作流程 / 类图
```
Client ──▶ HandlerA ──next──▶ HandlerB ──next──▶ HandlerC
```

### 3.3.4 代码示例
```java
// 抽象处理者（审批流程）
abstract class Approver {
    protected Approver next;
    public Approver setNext(Approver next) { this.next = next; return next; }
    public abstract void approve(int amount);
}

// 组长：审批 ≤1000
class TeamLeader extends Approver {
    @Override
    public void approve(int amount) {
        if (amount <= 1000) {
            System.out.println("组长批准：" + amount + "元");
        } else if (next != null) {
            next.approve(amount);
        }
    }
}

// 经理：审批 ≤5000
class Manager extends Approver {
    @Override
    public void approve(int amount) {
        if (amount <= 5000) {
            System.out.println("经理批准：" + amount + "元");
        } else if (next != null) {
            next.approve(amount);
        }
    }
}

// CEO：审批任意金额
class CEO extends Approver {
    @Override
    public void approve(int amount) {
        System.out.println("CEO批准：" + amount + "元");
    }
}

// 客户端：组装链，然后提交请求
public class Client {
    public static void main(String[] args) {
        Approver leader = new TeamLeader();
        Approver manager = new Manager();
        Approver ceo = new CEO();
        leader.setNext(manager).setNext(ceo); // 链式组装

        leader.approve(800);   // 组长批准：800元
        leader.approve(3000);  // 经理批准：3000元
        leader.approve(10000); // CEO批准：10000元
    }
}
```

### 3.3.5 优缺点
**优点：**
- 请求发送者与处理者解耦。
- 动态调整链的顺序和成员。
- 符合OCP，新增处理者无需修改现有代码。

**缺点：**
- 链过长时性能下降，调试困难。
- 请求可能不被任何处理者处理（需要有兜底处理）。

### 3.3.6 应用场景
- 多级审批流程。
- 请求拦截/过滤（权限、日志、限流）。
- 事件处理（多个监听器依次响应）。

### 3.3.7 经典框架实现 / 源码剖析
**Servlet Filter 责任链：**

- `Filter` 接口 → 抽象处理者
- 用户自定义 Filter → 具体处理者
- `FilterChain.doFilter()` → 传递请求给下一个节点

```java
public class LogFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        System.out.println("前置：记录请求日志");
        chain.doFilter(request, response); // 传递给下一个 Filter 或 Servlet
        System.out.println("后置：记录响应日志");
    }
}
```

**与经典责任链的差异：**
- 经典责任链：某节点处理后可中止链（单向传递）。
- Filter 责任链：强制双向（请求前 + 响应后都可处理），且通过 `chain.doFilter()` 确保请求到达 Servlet。

**Spring Security 的过滤器链（SecurityFilterChain）** 也是基于同样的 Servlet Filter 责任链机制实现的。

## 3.4 观察者设计模式

### 3.4.1 模式定义 / 概述
观察者模式（Observer Pattern）定义对象间的一种一对多的依赖关系：当一个对象（被观察者/主题）的状态发生变化时，所有依赖它的对象（观察者）都会自动得到通知并更新。属于**行为型**设计模式。

> **注**：与分布式系统中的发布-订阅模式（Pub-Sub）思想类似，但 GoF 中称为观察者模式。两者区别在于：观察者模式中 Subject 直接持有 Observer 引用；发布-订阅模式则通过独立的消息总线/事件通道完全解耦发布者和订阅者。

### 3.4.2 核心角色 / 结构组成
- **Subject（主题/被观察者）**：维护观察者列表，提供注册/注销/通知接口。
- **ConcreteSubject（具体主题）**：状态发生变化时调用 `notifyObservers()`。
- **Observer（观察者接口）**：定义 `update()` 方法。
- **ConcreteObserver（具体观察者）**：实现 `update()`，响应主题变化。

### 3.4.3 工作流程 / 类图
```
ConcreteSubject（状态变化）
    │
    └── notifyObservers() ──▶ ObserverA.update()
                         ──▶ ObserverB.update()
                         ──▶ ObserverC.update()
```

### 3.4.4 代码示例
```java
import java.util.ArrayList;
import java.util.List;

// 观察者接口
public interface Observer {
    void update(String event, Object data);
}

// 主题接口
public interface Subject {
    void addObserver(Observer observer);
    void removeObserver(Observer observer);
    void notifyObservers(String event, Object data);
}

// 具体主题：订单系统
public class OrderSystem implements Subject {
    private final List<Observer> observers = new ArrayList<>();

    @Override
    public void addObserver(Observer observer) { observers.add(observer); }
    @Override
    public void removeObserver(Observer observer) { observers.remove(observer); }
    @Override
    public void notifyObservers(String event, Object data) {
        observers.forEach(o -> o.update(event, data));
    }

    // 业务方法：创建订单后通知所有观察者
    public void createOrder(String orderId) {
        System.out.println("订单 " + orderId + " 创建成功");
        notifyObservers("ORDER_CREATED", orderId);
    }
}

// 具体观察者A：发送短信
public class SmsNotifier implements Observer {
    @Override
    public void update(String event, Object data) {
        if ("ORDER_CREATED".equals(event)) {
            System.out.println("【短信服务】发送订单确认短信，订单号：" + data);
        }
    }
}

// 具体观察者B：扣减库存
public class InventoryService implements Observer {
    @Override
    public void update(String event, Object data) {
        if ("ORDER_CREATED".equals(event)) {
            System.out.println("【库存服务】扣减库存，订单号：" + data);
        }
    }
}

// 客户端
public class Client {
    public static void main(String[] args) {
        OrderSystem orderSystem = new OrderSystem();
        orderSystem.addObserver(new SmsNotifier());
        orderSystem.addObserver(new InventoryService());

        orderSystem.createOrder("ORD-20261015-001");
        // 输出：
        // 订单 ORD-20261015-001 创建成功
        // 【短信服务】发送订单确认短信，订单号：ORD-20261015-001
        // 【库存服务】扣减库存，订单号：ORD-20261015-001
    }
}
```

### 3.4.5 优缺点
**优点：**
- 主题和观察者松耦合，互不依赖具体实现。
- 动态添加/删除观察者。
- 符合OCP。

**缺点：**
- 观察者过多时，通知遍历耗时。
- 观察者之间有执行顺序依赖时难以控制。
- 可能引发循环通知（A 通知 B，B 又触发 A）。

### 3.4.6 应用场景
- 事件驱动系统（GUI 事件监听）。
- 消息/事件总线。
- 数据变更通知（MVC 中 Model 变化通知 View）。

### 3.4.7 经典框架实现 / 源码剖析
**Spring 的 ApplicationEvent 机制（观察者模式）：**

Spring 的事件机制是对观察者模式的标准实现：

```java
// 1. 定义事件（相当于"通知数据"）
public class OrderCreatedEvent extends ApplicationEvent {
    private final String orderId;
    public OrderCreatedEvent(Object source, String orderId) {
        super(source);
        this.orderId = orderId;
    }
    public String getOrderId() { return orderId; }
}

// 2. 发布事件（具体主题调用 publishEvent）
@Service
public class OrderService {
    @Autowired
    private ApplicationEventPublisher publisher;

    public void createOrder(String orderId) {
        // 业务逻辑...
        publisher.publishEvent(new OrderCreatedEvent(this, orderId)); // 通知所有监听器
    }
}

// 3. 监听事件（具体观察者）
@Component
public class SmsListener implements ApplicationListener<OrderCreatedEvent> {
    @Override
    public void onApplicationEvent(OrderCreatedEvent event) {
        System.out.println("发送短信：" + event.getOrderId());
    }
}
```

在 `AbstractApplicationContext.refresh()` 中，容器本身也会在启动、刷新、关闭等节点发布内置事件（`ContextRefreshedEvent`、`ContextClosedEvent`）通知所有监听器。

## 3.5 迭代器设计模式

### 3.5.1 模式定义 / 概述
迭代器模式（Iterator Pattern）提供一种方法顺序访问一个聚合对象中的各个元素，而无需暴露该对象的内部表示。属于**行为型**设计模式。

> **注**：迭代器模式在 GoF 原著中英文为 Iterator Pattern，中文行业通用译名为"迭代器模式"。

### 3.5.2 核心角色 / 结构组成
- **Iterator（迭代器接口）**：定义 `hasNext()`、`next()` 等遍历接口。
- **ConcreteIterator（具体迭代器）**：实现遍历逻辑，维护当前位置。
- **Aggregate（聚合接口）**：定义 `iterator()` 方法，返回迭代器。
- **ConcreteAggregate（具体聚合）**：实际存储数据，创建迭代器。

### 3.5.3 工作流程 / 类图
```
Client ──▶ Aggregate ──iterator()──▶ Iterator
                                         ├── hasNext()
                                         └── next()
```

### 3.5.4 代码示例
```java
// 迭代器接口
public interface Iterator<T> {
    boolean hasNext();
    T next();
}

// 聚合接口
public interface Container<T> {
    Iterator<T> iterator();
    void add(T item);
}

// 具体聚合：书架
public class BookShelf implements Container<String> {
    private final List<String> books = new ArrayList<>();

    @Override
    public void add(String book) { books.add(book); }

    @Override
    public Iterator<String> iterator() {
        return new BookShelfIterator();
    }

    // 具体迭代器（内部类）
    private class BookShelfIterator implements Iterator<String> {
        private int index = 0;
        @Override
        public boolean hasNext() { return index < books.size(); }
        @Override
        public String next() { return books.get(index++); }
    }
}

// 客户端：无需关心内部存储结构
public class Client {
    public static void main(String[] args) {
        BookShelf shelf = new BookShelf();
        shelf.add("设计模式");
        shelf.add("重构");
        shelf.add("代码整洁之道");

        Iterator<String> it = shelf.iterator();
        while (it.hasNext()) {
            System.out.println(it.next());
        }
    }
}
```

### 3.5.5 优缺点
**优点：**
- 分离聚合对象的遍历行为，简化聚合类。
- 支持多种遍历方式（正序、逆序、过滤遍历）。
- 对客户端屏蔽内部存储结构。

**缺点：**
- 对于简单集合，引入迭代器增加了类数量。
- 迭代器与聚合对象需要同步更新。
