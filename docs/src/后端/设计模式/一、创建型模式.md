# 设计模式总结

# 一、创建型模式

## 1.1 单例设计模式

### 1.1.1 模式定义 / 概述
单例模式（Singleton Pattern）确保一个类只有一个实例，并提供一个全局访问点。属于**创建型**设计模式。

核心思想：控制实例数量，避免资源浪费；同时提供统一的访问方式。

### 1.1.2 核心角色 / 结构组成
- **Singleton 类**：持有自身的唯一实例，将构造方法私有化，对外提供静态的 `getInstance()` 方法。

### 1.1.3 工作流程 / 类图
```
Client ──getInstance()──▶ Singleton
                              │
                              ├── private static instance
                              ├── private Singleton()
                              └── public static getInstance()
```

### 1.1.4 代码示例

**方式一：饿汉式（线程安全，类加载时初始化）**
```java
public class Singleton {
    // 类加载时直接创建实例
    private static final Singleton INSTANCE = new Singleton();

    private Singleton() {}

    public static Singleton getInstance() {
        return INSTANCE;
    }
}
```

**方式二：懒汉式 + 双重检查锁（DCL，推荐）**
```java
public class Singleton {
    // volatile 防止指令重排序
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {                      // 第一次检查（无锁快速返回）
            synchronized (Singleton.class) {
                if (instance == null) {              // 第二次检查（防止重复创建）
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

**方式三：静态内部类（推荐，延迟加载 + 线程安全）**
```java
public class Singleton {
    private Singleton() {}

    // 只有在第一次调用 getInstance() 时，JVM 才加载 Holder 类
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

**方式四：枚举（最简洁，防反射攻击）**
```java
public enum Singleton {
    INSTANCE;

    public void doSomething() {
        System.out.println("Singleton via Enum");
    }
}
// 使用：Singleton.INSTANCE.doSomething();
```

### 1.1.5 优缺点
**优点：**
- 内存中只有一个实例，减少系统开销。
- 避免对资源（如数据库连接池）的多重占用。

**缺点：**

- 扩展困难，违背开闭原则（类本身控制自己的生命周期，职责也不单一）。
- 并发环境下实现复杂，容易出 bug。
- 单元测试困难（全局状态难以隔离）。

### 1.1.6 应用场景
- 线程池、数据库连接池
- 日志记录器
- 配置管理类
- Spring 的 Bean（默认 scope 就是单例）

### 1.1.7 经典框架实现
**Spring 中的双重检查锁单例**

Spring IoC 容器在获取单例 Bean 时使用了 DCL（双重检查锁）机制，核心代码位于 `DefaultSingletonBeanRegistry`：

```java
// Spring 源码简化版（DefaultSingletonBeanRegistry#getSingleton）
// ⚠️ 简化版，仅示意思路——实际 Spring 源码远比此处复杂
public Object getSingleton(String beanName, ObjectFactory<?> singletonFactory) {
    synchronized (this.singletonObjects) {
        Object singletonObject = this.singletonObjects.get(beanName);
        if (singletonObject == null) {
            // 前置检查、状态标记...
            singletonObject = singletonFactory.getObject(); // 创建 Bean
            addSingleton(beanName, singletonObject);        // 放入一级缓存
        }
        return singletonObject;
    }
}
```

Spring 通过 `singletonObjects`（一级缓存）、`earlySingletonObjects`（二级缓存）、`singletonFactories`（三级缓存）三级缓存机制，在保证单例的同时解决了循环依赖问题。

## 1.2 工厂方法设计模式

### 1.2.1 模式定义 / 概述
工厂方法模式（Factory Method Pattern）定义一个创建对象的接口，让子类决定实例化哪一个类，使一个类的实例化延迟到其子类。属于**创建型**设计模式。

工厂模式有三种形态：
- **简单工厂模式**（非GoF）：一个工厂类统一创建所有产品，不符合OCP。
- **工厂方法模式**（GoF）：每种产品对应一个工厂类，符合OCP。
- **抽象工厂模式**（GoF）：一个工厂类负责创建多个产品族，更高层次的抽象。

### 1.2.2 核心角色 / 结构组成
- **抽象工厂（Creator）**：声明工厂方法，返回抽象产品。
- **具体工厂（ConcreteCreator）**：实现工厂方法，返回具体产品。
- **抽象产品（Product）**：产品的公共接口。
- **具体产品（ConcreteProduct）**：实际被创建的对象。

### 1.2.3 工作流程 / 类图
```
Client ──▶ WeaponFactory(接口)
               ├── GunFactory ──▶ Gun
               ├── FighterFactory ──▶ Fighter
               └── DaggerFactory ──▶ Dagger
```

### 1.2.4 代码示例

**简单工厂（非GoF，了解即可）：**
```java
// 抽象产品
public abstract class Weapon {
    public abstract void attack();
}

// 具体产品
public class Tank extends Weapon {
    @Override
    public void attack() { System.out.println("坦克开炮！"); }
}
public class Fighter extends Weapon {
    @Override
    public void attack() { System.out.println("战斗机投下原子弹！"); }
}

// 工厂类（缺点：扩展时需要修改此类，违背OCP）
public class WeaponFactory {
    public static Weapon get(String weaponType) {
        return switch (weaponType) {
            case "TANK"    -> new Tank();
            case "FIGHTER" -> new Fighter();
            default        -> throw new RuntimeException("不支持该武器！");
        };
    }
}

// 客户端
public class Client {
    public static void main(String[] args) {
        Weapon w = WeaponFactory.get("TANK");
        w.attack(); // 坦克开炮！
    }
}
```

**工厂方法（GoF，推荐）：**
```java
// 抽象产品
public abstract class Weapon {
    public abstract void attack();
}

// 具体产品
public class Gun extends Weapon {
    @Override
    public void attack() { System.out.println("开枪射击！"); }
}
public class Dagger extends Weapon {
    @Override
    public void attack() { System.out.println("砍丫的！"); }
}

// 抽象工厂
public interface WeaponFactory {
    Weapon get();
}

// 具体工厂（扩展只需新增类，无需修改现有代码，符合OCP）
public class GunFactory implements WeaponFactory {
    @Override
    public Weapon get() { return new Gun(); }
}
public class DaggerFactory implements WeaponFactory {
    @Override
    public Weapon get() { return new Dagger(); }
}

// 客户端
public class Client {
    public static void main(String[] args) {
        WeaponFactory factory = new GunFactory();
        factory.get().attack(); // 开枪射击！
    }
}
```

### 1.2.5 优缺点
**优点：**
- 符合OCP开闭原则，扩展新产品只需增加工厂类，无需修改现有代码。
- 屏蔽产品的具体实现，调用者只关心接口。

**缺点：**
- 每增加一个产品，都需要同步增加一个工厂类，类数量成倍增加。

### 1.2.6 应用场景
- 客户端不知道需要创建哪种具体对象时。
- 系统需要独立于产品的创建、组合和表示时。

### 1.2.7 经典框架实现
**Spring 中的 BeanFactory 与简单工厂**

`BeanFactory` 的 `getBean(String name)` 方法从表层来看像简单工厂：根据名字返回对象。但本质上 `BeanFactory` 是功能极其丰富的 IoC 容器，远超简单工厂的职责。

**Spring 中的 FactoryBean 与工厂方法模式**

`FactoryBean` 是典型的工厂方法模式：
```java
// Spring FactoryBean 接口 —— 抽象工厂
public interface FactoryBean<T> {
    T getObject() throws Exception;         // 工厂方法
    Class<?> getObjectType();
    default boolean isSingleton() { return true; }
}

// 自定义具体工厂
public class MyConnectionFactory implements FactoryBean<Connection> {
    @Override
    public Connection getObject() throws Exception {
        return DriverManager.getConnection("jdbc:mysql://localhost/db", "root", "123");
    }
    @Override
    public Class<?> getObjectType() { return Connection.class; }
}
```
通过 `FactoryBean`，Spring 可以将复杂对象的创建逻辑封装到工厂内部，客户端调用 `getBean("myConnectionFactory")` 得到的是 `getObject()` 返回的对象。

## 1.3 抽象工厂设计模式

### 1.3.1 模式定义 / 概述
抽象工厂模式（Abstract Factory Pattern）提供一个创建一系列相关或相互依赖对象的接口，而无需指定具体类。属于**创建型**设计模式。

与工厂方法模式的区别：工厂方法针对**一个产品等级**，抽象工厂针对**多个产品等级**（即产品族）。

### 1.3.2 核心角色 / 结构组成
- **抽象工厂（AbstractFactory）**：声明创建多类产品的接口。
- **具体工厂（ConcreteFactory）**：实现同一产品族的所有产品创建。
- **抽象产品（AbstractProduct）**：每类产品的接口。
- **具体产品（ConcreteProduct）**：工厂创建的实际对象。

### 1.3.3 工作流程 / 类图
```
AbstractFactory
  ├── WeaponFactory ── 创建 Gun / Dagger
  └── FruitFactory  ── 创建 Apple / Orange
```

### 1.3.4 代码示例
```java
// 产品族1：武器
abstract class Weapon { public abstract void attack(); }
class Gun extends Weapon {
    @Override public void attack() { System.out.println("开枪射击！"); }
}
class Dagger extends Weapon {
    @Override public void attack() { System.out.println("砍丫的！"); }
}

// 产品族2：水果
abstract class Fruit { public abstract void ripeCycle(); }
class Apple extends Fruit {
    @Override public void ripeCycle() { System.out.println("苹果：8个月成熟"); }
}
class Orange extends Fruit {
    @Override public void ripeCycle() { System.out.println("橘子：10个月成熟"); }
}

// 抽象工厂（覆盖所有产品族）
abstract class AbstractFactory {
    public abstract Weapon getWeapon(String type);
    public abstract Fruit getFruit(String type);
}

// 武器族工厂
class WeaponFactory extends AbstractFactory {
    @Override
    public Weapon getWeapon(String type) {
        return switch (type) {
            case "Gun"    -> new Gun();
            case "Dagger" -> new Dagger();
            default       -> throw new RuntimeException("不支持");
        };
    }
    @Override public Fruit getFruit(String type) { return null; }
}

// 水果族工厂
class FruitFactory extends AbstractFactory {
    @Override public Weapon getWeapon(String type) { return null; }
    @Override
    public Fruit getFruit(String type) {
        return switch (type) {
            case "Apple"  -> new Apple();
            case "Orange" -> new Orange();
            default       -> throw new RuntimeException("不支持");
        };
    }
}

// 客户端
public class Client {
    public static void main(String[] args) {
        AbstractFactory wf = new WeaponFactory();
        wf.getWeapon("Gun").attack();    // 开枪射击！
        wf.getWeapon("Dagger").attack(); // 砍丫的！

        AbstractFactory ff = new FruitFactory();
        ff.getFruit("Apple").ripeCycle();  // 苹果：8个月成熟
        ff.getFruit("Orange").ripeCycle(); // 橘子：10个月成熟
    }
}
```

### 1.3.5 优缺点
**优点：**
- 保证客户端始终使用同一产品族的对象，避免产品错配。
- 易于切换产品族，只需切换工厂实现。

**缺点：**
- 扩展产品族困难，需要修改抽象工厂接口及所有具体工厂，违背OCP。

### 1.3.6 应用场景
- 系统需要独立于产品的创建和组合（如跨平台UI组件库）。
- 强调一系列对象配套使用的场景。

### 1.3.7 经典框架实现
- **MyBatis** 的 `SqlSessionFactory`：根据配置文件选择不同数据库实现（MySQL/Oracle/H2）时，内部产品（Statement、ResultSet 处理器）成套切换。
- **Spring** 的 `ApplicationContext`：`ClassPathXmlApplicationContext`、`AnnotationConfigApplicationContext` 等是具体工厂，提供不同环境下的 Bean、环境变量、资源加载等一整套产品族。

## 1.4 建造者设计模式

### 1.4.1 模式定义 / 概述
建造者模式（Builder Pattern）将一个复杂对象的构建过程与其表示分离，使得同样的构建过程可以创建不同的表示。属于**创建型**设计模式。

核心思想：把「如何一步步构建对象」的逻辑从对象本身中分离出来，对外提供链式或分步的建造接口。

### 1.4.2 核心角色 / 结构组成
- **Builder（抽象建造者）**：声明构建各部件的接口及返回产品的方法。
- **ConcreteBuilder（具体建造者）**：实现各部件的具体构建。
- **Director（指挥者）**：调用建造者来完成产品的构建流程（可选）。
- **Product（产品）**：最终被构建出来的复杂对象。

### 1.4.3 工作流程 / 类图
```
Director ──▶ Builder(接口)
                 └── ConcreteBuilder ──构建──▶ Product
```

### 1.4.4 代码示例

**方式一：Fluent Builder（静态内部类链式写法）：**
```java
// 产品
public class Computer {
    private String cpu;
    private String memory;
    private String disk;
    private String gpu;

    // 只有 Builder 才能设置属性
    private Computer() {}

    @Override
    public String toString() {
        return "Computer{cpu='" + cpu + "', memory='" + memory
             + "', disk='" + disk + "', gpu='" + gpu + "'}";
    }

    // 建造者（静态内部类，链式调用）
    public static class Builder {
        private final Computer computer = new Computer();

        public Builder cpu(String cpu) {
            computer.cpu = cpu;
            return this;
        }
        public Builder memory(String memory) {
            computer.memory = memory;
            return this;
        }
        public Builder disk(String disk) {
            computer.disk = disk;
            return this;
        }
        public Builder gpu(String gpu) {
            computer.gpu = gpu;
            return this;
        }
        public Computer build() {
            return computer;
        }
    }
}

// 客户端（链式调用）
public class Client {
    public static void main(String[] args) {
        Computer computer = new Computer.Builder()
                .cpu("Intel i9-13900K")
                .memory("64GB DDR5")
                .disk("2TB NVMe SSD")
                .gpu("RTX 4090")
                .build();
        System.out.println(computer);
    }
}
```

> **注**：GoF 原著中的经典建造者模式包含独立的 `Director` 类来控制构建流程，上述写法是 Java 中更常见的简化形式（省略了 Director），称为 Fluent Builder。

### 1.4.5 优缺点
**优点：**
- 封装性好，客户端不需要知道产品内部的构建细节。
- 建造步骤灵活，可以细粒度控制构建过程。
- 链式调用，代码可读性强。

**缺点：**
- 产品必须有共同点，范围受限。
- 若内部变化复杂，会有大量 Builder 子类。

### 1.4.6 应用场景
- 创建复杂对象，且对象的各部分属性相互依赖（如 SQL 构建器、请求报文构建）。
- 需要生成不同表示的同一对象（如不同格式的文档）。

### 1.4.7 经典框架实现
- **Lombok 的 `@Builder`**：自动生成 Builder 内部类，实现链式构建。
- **MyBatis 的 `SqlSessionFactoryBuilder`**：读取配置文件，一步步构建 `SqlSessionFactory`。
- **Spring Security 的 `HttpSecurity`**：通过链式配置构建安全规则，如 `.authorizeRequests().antMatchers(...).permitAll().and().formLogin()...`。
- **Java 标准库 `StringBuilder`**：本质是建造者，逐步 append 内容后 build 出字符串。

## 1.5 原型设计模式

### 1.5.1 模式定义 / 概述
原型模式（Prototype Pattern）通过复制（克隆）一个已有对象来创建新对象，而不是通过 new 关键字调用构造方法。属于**创建型**设计模式。

核心思想：当创建对象的代价较高（如需要复杂初始化）时，用一个原型对象为模板，快速克隆出新实例。

### 1.5.2 核心角色 / 结构组成
- **Prototype（原型接口）**：声明 `clone()` 方法。
- **ConcretePrototype（具体原型）**：实现克隆操作，返回自身的副本。
- **Client（客户端）**：通过调用 `clone()` 创建新对象。

### 1.5.3 工作流程 / 类图
```
Client ──clone()──▶ ConcretePrototype
                         │
                         └── 返回自身副本（浅拷贝 or 深拷贝）
```

### 1.5.4 代码示例

**浅拷贝（实现 Cloneable 接口）：**
```java
public class Sheep implements Cloneable {
    private String name;
    private int age;
    private Sheep friend; // 引用类型

    public Sheep(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // 浅拷贝：引用类型共享同一对象
    @Override
    public Sheep clone() {
        try {
            return (Sheep) super.clone();
        } catch (CloneNotSupportedException e) {
            throw new RuntimeException(e);
        }
    }

    public void setFriend(Sheep friend) { this.friend = friend; }
    public Sheep getFriend() { return friend; }
    public String getName() { return name; }
}

// 深拷贝（手动克隆引用类型）：
public class SheepDeep implements Cloneable {
    private String name;
    private SheepDeep friend;

    public SheepDeep(String name) { this.name = name; }
    public void setFriend(SheepDeep friend) { this.friend = friend; }

    @Override
    public SheepDeep clone() {
        try {
            SheepDeep copy = (SheepDeep) super.clone();
            // 深拷贝：对引用类型也调用 clone
            if (this.friend != null) {
                copy.friend = this.friend.clone();
            }
            return copy;
        } catch (CloneNotSupportedException e) {
            throw new RuntimeException(e);
        }
    }
}

public class Client {
    public static void main(String[] args) {
        Sheep original = new Sheep("多利", 3);
        Sheep clone1 = original.clone();
        Sheep clone2 = original.clone();
        System.out.println(original == clone1); // false，不同对象
        System.out.println(clone1.getName());   // 多利
    }
}
```

### 1.5.5 优缺点
**优点：**
- 性能高，克隆比 new + 初始化快（尤其是初始化代价高时）。
- 逃脱构造方法的限制，灵活复制对象。

**缺点：**
- 深拷贝实现较复杂，对象存在多层嵌套引用时需要递归克隆。
- 克隆对象与原对象完全独立，如果共享状态（浅拷贝）可能引发意外修改。

### 1.5.6 应用场景
- 对象初始化非常耗时（如需要读取数据库、网络请求）。
- 需要产生大量相似对象。
- 需要保存对象在某一时刻的状态（快照，与备忘录模式结合使用）。

### 1.5.7 经典框架实现
- **Spring 的 `prototype` scope Bean**：每次 `getBean()` 都通过反射创建新实例（不等同于 `Object.clone()`）。
- **Java 标准库**：`Object.clone()`、`ArrayList.clone()`、`HashMap.clone()` 均基于原型模式。
