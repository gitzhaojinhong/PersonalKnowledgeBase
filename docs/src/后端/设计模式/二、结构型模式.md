# 二、结构型模式

## 2.1 代理设计模式

### 2.1.1 模式定义 / 概述
代理模式（Proxy Pattern）为其他对象提供一种代理以控制对这个对象的访问。属于**结构型**设计模式。

代理对象与目标对象实现相同的接口，客户端无感知地使用代理对象，代理对象在调用目标对象的同时，可以在前后添加增强逻辑。

### 2.1.2 核心角色 / 结构组成
- **抽象主题（Subject）**：代理类和目标类的公共接口。
- **目标类（RealSubject）**：真正执行业务的对象。
- **代理类（Proxy）**：持有对目标类的引用，在调用目标方法时增加额外逻辑。

### 2.1.3 工作流程 / 类图
```
Client ──▶ Subject(接口)
               ├── RealSubject（目标）
               └── Proxy（代理）──持有──▶ RealSubject
```

代理模式分两种实现形式：
- **静态代理**：手动编写代理类，一接口一代理，类数量多。
- **动态代理**：运行期在内存中动态生成代理字节码，一个处理器覆盖所有接口。

### 2.1.4 代码示例

**静态代理：**
```java
// 公共接口
public interface OrderService {
    void generate();
    void detail();
    void modify();
}

// 目标类
public class OrderServiceImpl implements OrderService {
    @Override
    public void generate() {
        try { Thread.sleep(1234); } catch (InterruptedException e) { e.printStackTrace(); }
        System.out.println("订单已生成");
    }
    @Override
    public void detail() {
        try { Thread.sleep(2541); } catch (InterruptedException e) { e.printStackTrace(); }
        System.out.println("订单信息如下：******");
    }
    @Override
    public void modify() {
        try { Thread.sleep(1010); } catch (InterruptedException e) { e.printStackTrace(); }
        System.out.println("订单已修改");
    }
}

// 静态代理类（统计耗时，不修改目标类，符合OCP）
public class OrderServiceProxy implements OrderService {
    private final OrderService target;
    public OrderServiceProxy(OrderService target) { this.target = target; }

    @Override
    public void generate() {
        long begin = System.currentTimeMillis();
        target.generate();
        System.out.println("generate耗时：" + (System.currentTimeMillis() - begin) + "ms");
    }
    @Override
    public void detail() {
        long begin = System.currentTimeMillis();
        target.detail();
        System.out.println("detail耗时：" + (System.currentTimeMillis() - begin) + "ms");
    }
    @Override
    public void modify() {
        long begin = System.currentTimeMillis();
        target.modify();
        System.out.println("modify耗时：" + (System.currentTimeMillis() - begin) + "ms");
    }
}
```

**JDK 动态代理：**
```java
import java.lang.reflect.InvocationHandler;
import java.lang.reflect.Method;
import java.lang.reflect.Proxy;

// 调用处理器（写一次，适用所有接口）
public class TimerInvocationHandler implements InvocationHandler {
    private final Object target;
    public TimerInvocationHandler(Object target) { this.target = target; }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        long begin = System.currentTimeMillis();
        Object retValue = method.invoke(target, args); // 调用目标方法
        System.out.println(method.getName() + "耗时：" + (System.currentTimeMillis() - begin) + "ms");
        return retValue;
    }
}

// 客户端
public class Client {
    public static void main(String[] args) {
        OrderService target = new OrderServiceImpl();
        // 动态生成代理对象（只需目标对象，无需手写代理类）
        OrderService proxy = (OrderService) Proxy.newProxyInstance(
            target.getClass().getClassLoader(),
            target.getClass().getInterfaces(),
            new TimerInvocationHandler(target)
        );
        proxy.generate();
        proxy.detail();
        proxy.modify();
    }
}
```

**CGLIB 动态代理（无需接口）：**
```java
import net.sf.cglib.proxy.Enhancer;
import net.sf.cglib.proxy.MethodInterceptor;
import net.sf.cglib.proxy.MethodProxy;
import java.lang.reflect.Method;

public class UserService {
    public void login()  { System.out.println("用户正在登录系统...."); }
    public void logout() { System.out.println("用户正在退出系统...."); }
}

public class TimerMethodInterceptor implements MethodInterceptor {
    @Override
    public Object intercept(Object proxy, Method method, Object[] objects, MethodProxy methodProxy) throws Throwable {
        long begin = System.currentTimeMillis();
        Object retValue = methodProxy.invokeSuper(proxy, objects); // 直接字节码调用，性能更高
        System.out.println(method.getName() + "耗时：" + (System.currentTimeMillis() - begin) + "ms");
        return retValue;
    }
}

public class Client {
    public static void main(String[] args) {
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(UserService.class);
        enhancer.setCallback(new TimerMethodInterceptor());
        UserService proxy = (UserService) enhancer.create();
        proxy.login();
        proxy.logout();
    }
}
```

> **注意：**
> 1. Spring Boot（1.4 起，包含 2.x/3.x）默认使用 CGLIB（`spring.aop.proxy-target-class=true`），可通过 `spring.aop.proxy-target-class=false` 改回 JDK；纯 Spring Framework 仍保持有接口时默认 JDK 动态代理。
> 2. 动态代理只能代理 `public` 方法。
> 3. CGLIB 基于继承，被代理类不能用 `final` 修饰。

### 2.1.5 优缺点
**优点：**
- 符合OCP，无需修改目标类即可增强功能。
- 动态代理实现代码复用，一个 Handler/Interceptor 覆盖所有目标。

**缺点：**
- 静态代理类数量多（类爆炸）。
- 动态代理引入反射，有少量性能损耗。

### 2.1.6 应用场景
- AOP（面向切面编程）：日志、事务、权限、性能监控。
- 远程代理（RPC 调用）。
- 虚拟代理（延迟加载大对象）。
- 保护代理（访问控制）。

### 2.1.7 经典框架实现 / 源码剖析
**Spring AOP**

Spring AOP 通过动态代理实现面向切面编程：
- 目标类实现了接口 → 使用 JDK 动态代理（`JdkDynamicAopProxy`）
- 目标类没有接口 → 使用 CGLIB（`CglibAopProxy`）

核心入口：`DefaultAopProxyFactory#createAopProxy()`

```java
// Spring 源码核心逻辑简化版（DefaultAopProxyFactory）
// 注：完整实现还处理了 Lambda 类、代理类、接口类等边界情况
public AopProxy createAopProxy(AdvisedSupport config) {
    if (config.isOptimize() || config.isProxyTargetClass()
            || hasNoUserSuppliedProxyInterfaces(config)) {
        return new ObjenesisCglibAopProxy(config); // CGLIB
    } else {
        return new JdkDynamicAopProxy(config);     // JDK（有接口时默认）
    }
}
```

## 2.2 装饰器设计模式

### 2.2.1 模式定义 / 概述
装饰器模式（Decorator Pattern）动态地给对象添加额外的职责，相比子类继承，装饰器模式更灵活。属于**结构型**设计模式。

核心思想：装饰器和目标对象实现同一接口，装饰器持有目标对象的引用，在调用目标方法的同时叠加新功能，且可以多层嵌套。

### 2.2.2 核心角色 / 结构组成
- **Component（抽象组件）**：定义对象接口。
- **ConcreteComponent（具体组件）**：被装饰的原始对象。
- **Decorator（抽象装饰器）**：持有 Component 引用，实现相同接口。
- **ConcreteDecorator（具体装饰器）**：添加具体功能。

### 2.2.3 工作流程 / 类图
```
Client ──▶ Component(接口)
               ├── ConcreteComponent（原始对象）
               └── Decorator（持有Component）
                       └── ConcreteDecorator（叠加新功能）
```

### 2.2.4 代码示例
```java
// 抽象组件
public interface Coffee {
    String getDescription();
    double getCost();
}

// 具体组件（原始咖啡）
public class SimpleCoffee implements Coffee {
    @Override
    public String getDescription() { return "简单咖啡"; }
    @Override
    public double getCost() { return 10.0; }
}

// 抽象装饰器
public abstract class CoffeeDecorator implements Coffee {
    protected final Coffee decoratedCoffee;
    public CoffeeDecorator(Coffee coffee) { this.decoratedCoffee = coffee; }
    @Override
    public String getDescription() { return decoratedCoffee.getDescription(); }
    @Override
    public double getCost() { return decoratedCoffee.getCost(); }
}

// 具体装饰器：加牛奶
public class MilkDecorator extends CoffeeDecorator {
    public MilkDecorator(Coffee coffee) { super(coffee); }
    @Override
    public String getDescription() { return super.getDescription() + " + 牛奶"; }
    @Override
    public double getCost() { return super.getCost() + 2.0; }
}

// 具体装饰器：加糖
public class SugarDecorator extends CoffeeDecorator {
    public SugarDecorator(Coffee coffee) { super(coffee); }
    @Override
    public String getDescription() { return super.getDescription() + " + 糖"; }
    @Override
    public double getCost() { return super.getCost() + 1.0; }
}

// 客户端：多层装饰，灵活叠加
public class Client {
    public static void main(String[] args) {
        Coffee coffee = new SimpleCoffee();
        coffee = new MilkDecorator(coffee);     // 加牛奶
        coffee = new SugarDecorator(coffee);    // 再加糖
        coffee = new MilkDecorator(coffee);     // 再加一份牛奶
        System.out.println(coffee.getDescription()); // 简单咖啡 + 牛奶 + 糖 + 牛奶
        System.out.println(coffee.getCost());         // 15.0
    }
}
```

### 2.2.5 优缺点
**优点：**
- 比继承更灵活，可以在运行时动态添加/撤销功能。
- 符合OCP，无需修改原有类即可扩展功能。
- 可以多层叠加，组合出丰富的功能。

**缺点：**
- 多层装饰会产生很多小对象，增加调试复杂度。
- 装饰层次过深时，代码可读性下降。

### 2.2.6 应用场景
- 不能/不想通过继承扩展功能时（如 final 类）。
- 需要动态（运行时）增删功能。
- 需要对多个对象分别独立添加不同功能组合。

### 2.2.7 经典框架实现 / 源码剖析
**Java IO 流（最经典的装饰器模式）：**
```java
// 多层装饰：FileInputStream → BufferedInputStream → DataInputStream
InputStream in = new DataInputStream(
                    new BufferedInputStream(
                        new FileInputStream("data.bin")));
```
`InputStream` 是抽象组件，`FileInputStream` 是具体组件，`FilterInputStream` 是抽象装饰器，`BufferedInputStream`/`DataInputStream` 是具体装饰器。

**Spring IoC 中的 BeanWrapper：**
```java
// Bean 的装饰器模式应用
BeanWrapper wrapper = new BeanWrapperImpl(targetObject);
// Spring 用 BeanWrapper 装饰 Bean，增加属性访问、类型转换等功能
```

**Spring 中类名带 `Decorator` 和 `Wrapper` 的类，均是装饰器模式的体现。**

## 2.3 适配器设计模式

### 2.3.1 模式定义 / 概述
适配器模式（Adapter Pattern）将一个类的接口转换成客户期望的另一个接口，使原本由于接口不兼容而不能一起工作的类可以协同工作。属于**结构型**设计模式。

适配器有两种实现：
- **对象适配器**：组合方式持有被适配对象（推荐）。
- **类适配器**：通过继承被适配类（耦合度高，较少使用）。

另有**缺省适配器**（Default Adapter）：为接口提供默认实现，子类按需覆盖。

### 2.3.2 核心角色 / 结构组成
- **Target（目标接口）**：客户端期望的接口。
- **Adaptee（被适配者）**：现有的、接口不兼容的类。
- **Adapter（适配器）**：实现 Target，内部持有 Adaptee，将 Target 的调用转发给 Adaptee。

### 2.3.3 工作流程 / 类图
```
Client ──▶ Target(接口) ◁── Adapter ──持有──▶ Adaptee
```

### 2.3.4 代码示例

**对象适配器（日志框架适配）：**
```java
// 目标接口
interface MyLogger {
    void log(String message);
}

// 被适配者1（方法名不同）
class Log4jLogger {
    public void logMessage(String msg) { System.out.println("Log4j: " + msg); }
}

// 被适配者2
class Slf4jLogger {
    public void log(String content) { System.out.println("SLF4J: " + content); }
}

// 适配器1
class Log4jAdapter implements MyLogger {
    private final Log4jLogger log4j;
    public Log4jAdapter(Log4jLogger log4j) { this.log4j = log4j; }
    @Override
    public void log(String message) { log4j.logMessage(message); }
}

// 适配器2
class Slf4jAdapter implements MyLogger {
    private final Slf4jLogger slf4j;
    public Slf4jAdapter(Slf4jLogger slf4j) { this.slf4j = slf4j; }
    @Override
    public void log(String message) { slf4j.log(message); }
}

public class Client {
    public static void main(String[] args) {
        MyLogger logger1 = new Log4jAdapter(new Log4jLogger());
        MyLogger logger2 = new Slf4jAdapter(new Slf4jLogger());
        logger1.log("使用Log4j记录日志");
        logger2.log("使用SLF4J记录日志");
    }
}
```

**缺省适配器（Default Adapter）：**
```java
// 接口方法过多，子类不想全部实现
interface ServiceInterface {
    void operation1();
    void operation2();
    void operation3();
}

// 缺省适配器：全部空实现
abstract class DefaultAdapter implements ServiceInterface {
    @Override public void operation1() {}
    @Override public void operation2() {}
    @Override public void operation3() {}
}

// 具体子类只覆盖感兴趣的方法
class ConcreteService extends DefaultAdapter {
    @Override
    public void operation2() { System.out.println("只实现了operation2方法"); }
}
```

### 2.3.5 优缺点
**优点：**
- 复用已有代码，不修改 Adaptee 即可让其适配新接口。
- 增加类的透明性，客户端对 Adaptee 无感知。

**缺点：**
- 适配器过多时代码混乱，阅读困难。
- 类适配器方式因 Java 单继承限制，有局限性。

### 2.3.6 应用场景
- 整合第三方库，其接口与系统接口不一致时。
- 旧系统改造，新旧接口需要兼容时。

### 2.3.7 经典框架实现 / 源码剖析
**GenericServlet（缺省适配器）**

`Servlet` 接口有 `init()`、`service()`、`destroy()`、`getServletConfig()`、`getServletInfo()` 多个方法。`GenericServlet` 作为抽象类提供了除 `service()` 之外所有方法的默认空实现，开发者继承后只需关注 `service()` 方法：

```java
public abstract class GenericServlet implements Servlet, ServletConfig, java.io.Serializable {
    private transient ServletConfig config;

    @Override
    public void init(ServletConfig config) throws ServletException {
        this.config = config;
        this.init(); // 模板方法：留给子类覆盖的无参init
    }

    @Override
    public abstract void service(ServletRequest req, ServletResponse res)
            throws ServletException, IOException;

    @Override
    public void destroy() {} // 默认空实现

    @Override
    public String getServletInfo() { return ""; } // 默认空实现

    public void init() throws ServletException {} // 方便子类重写的钩子
}
```

**Spring MVC 的 HandlerAdapter**

Spring MVC 使用适配器模式统一处理不同类型的 Handler（`@Controller`、`HttpRequestHandler`、`Servlet`）：
```java
// 目标接口
public interface HandlerAdapter {
    boolean supports(Object handler);
    ModelAndView handle(...) throws Exception;
}
// 具体适配器
RequestMappingHandlerAdapter  // 处理 @RequestMapping 方法
HttpRequestHandlerAdapter     // 处理 HttpRequestHandler
SimpleControllerHandlerAdapter // 处理 Controller 接口
```

## 2.4 组合设计模式

### 2.4.1 模式定义 / 概述
组合模式（Composite Pattern）将对象组织成树形结构以表示"部分-整体"的层次关系，使客户端对单个对象和组合对象的使用具有一致性。属于**结构型**设计模式。

### 2.4.2 核心角色 / 结构组成
- **Component（抽象组件）**：定义叶子节点和容器节点的公共接口。
- **Leaf（叶子节点）**：树的基本元素，没有子节点。
- **Composite（容器节点）**：包含子组件列表，实现子节点的管理操作。

### 2.4.3 工作流程 / 类图
```
Component(抽象)
    ├── Leaf（叶子，无子节点）
    └── Composite（容器，持有List<Component>）
            ├── Leaf
            ├── Leaf
            └── Composite
                    └── Leaf
```

### 2.4.4 代码示例
```java
// 抽象组件（文件系统）
public abstract class FileSystemNode {
    protected String name;
    public FileSystemNode(String name) { this.name = name; }
    public abstract long getSize();
    public abstract void print(String indent);
}

// 叶子节点：文件
public class File extends FileSystemNode {
    private long size;
    public File(String name, long size) {
        super(name);
        this.size = size;
    }
    @Override
    public long getSize() { return size; }
    @Override
    public void print(String indent) {
        System.out.println(indent + "📄 " + name + " (" + size + "KB)");
    }
}

// 容器节点：目录
public class Directory extends FileSystemNode {
    private final List<FileSystemNode> children = new ArrayList<>();

    public Directory(String name) { super(name); }

    public void add(FileSystemNode node) { children.add(node); }
    public void remove(FileSystemNode node) { children.remove(node); }

    @Override
    public long getSize() {
        return children.stream().mapToLong(FileSystemNode::getSize).sum();
    }
    @Override
    public void print(String indent) {
        System.out.println(indent + "📁 " + name);
        children.forEach(child -> child.print(indent + "  "));
    }
}

// 客户端：对目录和文件统一操作
public class Client {
    public static void main(String[] args) {
        Directory root = new Directory("root");
        Directory src = new Directory("src");
        src.add(new File("Main.java", 20));
        src.add(new File("Utils.java", 15));

        Directory test = new Directory("test");
        test.add(new File("MainTest.java", 10));

        root.add(src);
        root.add(test);
        root.add(new File("pom.xml", 5));

        root.print("");           // 打印树形结构
        System.out.println("总大小: " + root.getSize() + "KB"); // 50KB
    }
}
```

### 2.4.5 优缺点
**优点：**
- 客户端对叶子和容器统一处理，简化调用。
- 方便构建树形结构，扩展性好（增加新的叶子/容器类型无需修改现有代码）。

**缺点：**
- 设计过于抽象，在强类型语言中，要区分 Leaf 和 Composite 时较麻烦。
- 不能限制容器中的子节点类型（需额外校验）。

### 2.4.6 应用场景
- 文件系统（目录和文件）。
- XML / HTML / JSON DOM 树。
- 菜单和菜单项。
- 组织结构树（部门与员工）。

### 2.4.7 经典框架实现
- **MyBatis** 的 `SqlNode`（动态SQL）：`IfSqlNode`、`ChooseSqlNode`、`MixedSqlNode`（容器，包含多个 `SqlNode`）构成树形结构，统一由 `apply()` 方法渲染。
- **Spring Security** 的权限决策树：多个 `AccessDecisionVoter` 组合成 `AccessDecisionManager`。

## 2.5 享元设计模式

### 2.5.1 模式定义 / 概述
享元模式（Flyweight Pattern）运用共享技术有效地支持大量细粒度的对象，减少内存消耗。属于**结构型**设计模式。

核心思想：将对象的状态分为**内部状态**（不变，可共享）和**外部状态**（变化，由外部传入），将内部状态相同的对象共享同一个实例。

### 2.5.2 核心角色 / 结构组成
- **Flyweight（享元接口）**：定义享元对象的接口。
- **ConcreteFlyweight（具体享元）**：存储内部状态。
- **FlyweightFactory（享元工厂）**：管理享元对象的创建和缓存（通常用 Map）。
- **Client（客户端）**：维护外部状态，通过工厂获取享元对象。

### 2.5.3 工作流程 / 类图
```
Client ──▶ FlyweightFactory(Map缓存)
                  └── ConcreteFlyweight（共享实例）
                            内部状态（不变）
                            外部状态（由Client传入）
```

### 2.5.4 代码示例
```java
// 享元接口
public interface ChessPiece {
    void display(int x, int y); // x,y 是外部状态（位置）
}

// 具体享元：棋子（内部状态：颜色、名称；外部状态：坐标）
public class ConcreteChessPiece implements ChessPiece {
    private final String color;  // 内部状态（共享）
    private final String name;   // 内部状态（共享）

    public ConcreteChessPiece(String color, String name) {
        this.color = color;
        this.name = name;
    }
    @Override
    public void display(int x, int y) { // x,y 是外部状态，不存储
        System.out.println(color + "棋子[" + name + "] 位于 (" + x + "," + y + ")");
    }
}

// 享元工厂（缓存共享实例）
public class ChessPieceFactory {
    private static final Map<String, ChessPiece> pool = new HashMap<>();

    public static ChessPiece getChessPiece(String color, String name) {
        String key = color + "-" + name;
        return pool.computeIfAbsent(key, k -> {
            System.out.println("创建新享元对象：" + key);
            return new ConcreteChessPiece(color, name);
        });
    }
}

// 客户端
public class Client {
    public static void main(String[] args) {
        ChessPiece rook1 = ChessPieceFactory.getChessPiece("红", "车");
        ChessPiece rook2 = ChessPieceFactory.getChessPiece("红", "车");
        System.out.println(rook1 == rook2); // true，同一共享对象

        rook1.display(0, 0); // 红棋子[车] 位于 (0,0)
        rook2.display(0, 7); // 红棋子[车] 位于 (0,7)（外部状态不同，但对象相同）
    }
}
```

### 2.5.5 优缺点
**优点：**
- 大幅减少对象数量，节省内存（特别是大量细粒度对象场景）。

**缺点：**
- 需要分离内部/外部状态，增加设计复杂度。
- 外部状态由客户端维护，线程安全需特别注意。

### 2.5.6 应用场景
- 系统中存在大量相似对象，且内存消耗大时。
- 对象的大部分状态可以外部化。

### 2.5.7 经典框架实现
- **Java 的 `String` 常量池**：字符串字面量由 JVM 统一缓存，相同字面量共享同一实例。
- **Java 的 `Integer.valueOf(int)`**：`-128 ~ 127` 范围的整数共享缓存实例（`IntegerCache`）。
- **数据库连接池**：`Druid`、`HikariCP` 等通过复用连接对象节省创建销毁开销，与享元思想一致。

## 2.6 外观设计模式

### 2.6.1 模式定义 / 概述
外观模式（Facade Pattern）为子系统中的一组接口提供一个一致的高层接口，使子系统更容易使用。属于**结构型**设计模式。

核心思想：把复杂的子系统调用封装在一个 Facade 类中，对外只暴露简单的接口。

### 2.6.2 核心角色 / 结构组成
- **Facade（外观类）**：对外的统一入口，内部调用多个子系统。
- **SubSystem（子系统）**：实现具体业务功能的类，互相独立。
- **Client（客户端）**：只与 Facade 交互，不直接访问子系统。

### 2.6.3 工作流程 / 类图
```
Client ──▶ Facade
               ├── SubSystemA
               ├── SubSystemB
               └── SubSystemC
```

### 2.6.4 代码示例
```java
// 子系统1：CPU
class CPU {
    public void startup() { System.out.println("CPU 启动..."); }
    public void shutdown() { System.out.println("CPU 关闭..."); }
}

// 子系统2：内存
class Memory {
    public void startup() { System.out.println("内存 加载..."); }
    public void shutdown() { System.out.println("内存 释放..."); }
}

// 子系统3：硬盘
class HardDisk {
    public void startup() { System.out.println("硬盘 读取数据..."); }
    public void shutdown() { System.out.println("硬盘 写入数据..."); }
}

// 外观类（封装复杂的启动/关机流程）
public class ComputerFacade {
    private final CPU cpu = new CPU();
    private final Memory memory = new Memory();
    private final HardDisk disk = new HardDisk();

    public void startup() {
        System.out.println("=== 开机 ===");
        cpu.startup();
        memory.startup();
        disk.startup();
        System.out.println("=== 启动完成 ===");
    }

    public void shutdown() {
        System.out.println("=== 关机 ===");
        disk.shutdown();
        memory.shutdown();
        cpu.shutdown();
        System.out.println("=== 关机完成 ===");
    }
}

// 客户端：一键开机/关机，无需了解内部流程
public class Client {
    public static void main(String[] args) {
        ComputerFacade computer = new ComputerFacade();
        computer.startup();
        computer.shutdown();
    }
}
```

### 2.6.5 优缺点
**优点：**
- 降低客户端与子系统的耦合度，简化调用。
- 子系统可以独立变化，不影响客户端。

**缺点：**

- 不符合OCP，若需要扩展子系统功能，可能需要修改外观类。
- 外观类承担过多职责时会成为"上帝类"。

### 2.6.6 应用场景
- 为复杂子系统提供简单接口（如第三方库封装）。
- 分层架构中，Service 层对 DAO 层/多个外部系统的调用封装也体现了外观模式的思想。

### 2.6.7 经典框架实现
- **Spring** 的 `JdbcTemplate`：对 JDBC 繁琐的 `Connection`、`Statement`、`ResultSet` 操作进行封装，提供简洁的 API（同时体现了模板方法模式，见§14模板方法模式）。
- **SLF4J**：为不同日志实现（Log4j、Logback、JUL）提供统一的外观接口。
- **Tomcat** 的 `RequestFacade`：对底层 `Request` 对象进行包装，暴露给 Servlet 的 `HttpServletRequest` 接口。

## 2.7 桥接设计模式

### 2.7.1 模式定义 / 概述
桥接模式（Bridge Pattern）将抽象部分与其实现部分分离，使它们都可以独立变化。属于**结构型**设计模式。

核心思想：用聚合（组合）替代继承，让"抽象"和"实现"沿各自的维度独立扩展，避免继承层次爆炸。

### 2.7.2 核心角色 / 结构组成
- **Abstraction（抽象化）**：持有 Implementor 的引用，定义高层接口。
- **RefinedAbstraction（扩展抽象化）**：扩展 Abstraction 的功能。
- **Implementor（实现化接口）**：定义底层实现接口（不必与 Abstraction 接口一致）。
- **ConcreteImplementor（具体实现化）**：实现 Implementor 接口。

### 2.7.3 工作流程 / 类图
```
Abstraction ──持有──▶ Implementor(接口)
    └── RefinedAbstraction    └── ConcreteImplementorA
                              └── ConcreteImplementorB
```

### 2.7.4 代码示例
```java
// 实现化接口：消息发送渠道
public interface MessageSender {
    void send(String content);
}

// 具体实现A：短信
public class SMSSender implements MessageSender {
    @Override
    public void send(String content) {
        System.out.println("【短信】发送：" + content);
    }
}

// 具体实现B：邮件
public class EmailSender implements MessageSender {
    @Override
    public void send(String content) {
        System.out.println("【邮件】发送：" + content);
    }
}

// 抽象化：消息类型
public abstract class Message {
    protected MessageSender sender; // 桥接：持有实现化引用
    public Message(MessageSender sender) { this.sender = sender; }
    public abstract void sendMessage(String content);
}

// 扩展抽象化A：普通消息
public class NormalMessage extends Message {
    public NormalMessage(MessageSender sender) { super(sender); }
    @Override
    public void sendMessage(String content) {
        sender.send("[普通] " + content);
    }
}

// 扩展抽象化B：紧急消息
public class UrgentMessage extends Message {
    public UrgentMessage(MessageSender sender) { super(sender); }
    @Override
    public void sendMessage(String content) {
        sender.send("[紧急!!] " + content);
    }
}

// 客户端：抽象和实现独立组合，2×2=4种组合无需4个子类
public class Client {
    public static void main(String[] args) {
        Message msg1 = new NormalMessage(new SMSSender());
        msg1.sendMessage("会议通知");        // 【短信】发送：[普通] 会议通知

        Message msg2 = new UrgentMessage(new EmailSender());
        msg2.sendMessage("服务器宕机！");    // 【邮件】发送：[紧急!!] 服务器宕机！
    }
}
```

### 2.7.5 优缺点
**优点：**
- 抽象与实现分离，独立扩展，避免继承层次爆炸（M×N 类 → M+N 类）。
- 符合OCP，对扩展开放。

**缺点：**
- 增加了系统复杂度，对初学者不易理解。

### 2.7.6 应用场景
- 系统存在两个独立变化的维度，且都需要扩展。
- 不希望使用继承，或继承层次过多时。

### 2.7.7 经典框架实现
- **JDBC**：`java.sql.Driver` 接口是桥接的实现化，不同数据库驱动（MySQL、Oracle）实现了 `Driver` 接口；`Connection`、`Statement` 等是抽象化。
- **Java AWT/Swing**：`Component`（抽象）通过 `Peer`（实现）连接到不同操作系统的本地 GUI 控件。
