# 异步领券功能高并发优化方案

## 1.1 优化思路

### 1.1.1 原始串行流程的问题

领券业务的原始流程如下：

```
用户 → 查询优惠券(DB) → 校验发放时间 → 校验库存(DB) → 统计已领取数量(DB)
     → 校验限领数量 → 更新已发放数量(DB) → 新增用户券(DB) → 结束
```

整个流程存在三个核心问题：

- **多次数据库读写**：流程中包含查询、更新、新增共计 4~5 次数据库操作
- **接口串行加锁**：为保证线程安全需要加分布式锁，所有并发请求被迫排队
- **响应时间长**：串行执行导致接口 RT 随并发量线性增长，用户体验差

### 1.1.2 高并发写优化方向对比

对于高并发写业务，常见的两种优化思路如下：

**变同步为异步（MQ削峰）：**

用户请求只需发一条 MQ 消息，不等待业务执行完毕（20ms 级响应），由消费者异步完成耗时的数据库写操作。

**合并写请求（Redis缓存 + 批量落库）：**

将大量写请求先缓存到 Redis，通过定时任务定期批量写入数据库，将"每次请求写一次 DB"变为"每 N 秒批量写一次"。

**方案选型：**

| 对比维度 | 合并写请求 | 异步写（MQ） |
|----------|-----------|-------------|
| 适用场景 | 写频率高、写数据简单（如点赞数） | 业务链长、逻辑复杂 |
| 一致性 | 最终一致，有短暂延迟 | 最终一致，消费完成后写库 |
| 复杂度 | 适中 | 相对较高（需引入 MQ） |

> 领券业务逻辑长且复杂（资格校验 + 数据库多步写入），**适合选用异步写方案**。

### 1.1.3 优化迭代过程

**第一次迭代（直接异步，有缺陷）：**

用户请求 → 直接发 MQ → 立即返回。问题：资格校验还未完成就已返回，无法告知用户是否成功。

**第二次迭代（资格校验前置）：**

将资格校验逻辑提前到 MQ 发送前完成，校验通过再发消息。问题：资格校验仍需多次查数据库并加锁，效率提升不明显。

**第三次迭代（Redis缓存校验，最终方案）：**

将优惠券关键数据缓存到 Redis，资格校验全部命中缓存，彻底避免数据库查询，效率大幅提升。

## 1.2 方案设计

### 1.2.1 整体流程设计

优化后的最终流程分为两个阶段：**同步阶段（基于 Redis 的资格校验）** 和 **异步阶段（MQ 消费 + 数据库写入）**。

**直接领券（免兑换码）流程：**

```
用户请求
   ↓
[同步阶段 - Redis 资格校验]
   ├── 检查优惠券缓存是否存在
   ├── 校验发放时间（issueBeginTime / issueEndTime）
   ├── 校验库存（totalNum > 0）
   ├── 累加用户已领数量并校验限领数量（userLimit）
   └── 扣减 Redis 中的库存（HINCRBY totalNum -1）
   ↓
发送 MQ 消息（userId + couponId）
   ↓
立即返回"领取成功"
   ↓
[异步阶段 - MQ 消费写数据库]
   ├── 更新优惠券已发放数量 +1（DB）
   └── 新增用户券记录（DB）
```

**兑换码领券流程：**

```
用户请求（携带兑换码）
   ↓
[同步阶段 - Redis 资格校验]
   ├── 解析并验签兑换码，得到序列号
   ├── 校验是否已兑换（Redis BitMap GETBIT）
   ├── 通过序列号查找对应优惠券 id（Redis ZSet ZRANGEBYSCORE）
   ├── 校验优惠券是否存在及是否在发放时间内
   ├── 累加用户已领数量并校验限领数量
   └── 标记兑换码已使用（Redis BitMap SETBIT）
   ↓
发送 MQ 消息（userId + couponId + serialNum）
   ↓
立即返回"兑换成功"
   ↓
[异步阶段 - MQ 消费写数据库]
   ├── 更新优惠券已发放数量 +1（DB）
   ├── 新增用户券记录（DB）
   └── 更新兑换码状态为已兑换（DB）
```

### 1.2.2 Redis 数据结构设计

#### 1.2.2.1 优惠券基础信息缓存

**数据结构：Hash**

**选型理由：**

资格校验只需要 4 个字段（发放时间、库存、每人限领数），不需要存全量优惠券信息，节省内存。使用 Hash 结构，每个字段独立存储，可以通过 `HINCRBY` 命令原子地修改单个字段（如扣减库存），无需读-改-写三步。

**结构示意：**

```
KEY: prs:coupon:{couponId}   →   Hash
    field           value
    issueBeginTime  1679875200000（毫秒时间戳）
    issueEndTime    1682899200000（毫秒时间戳）
    totalNum        100（剩余库存，每次领取后 -1）
    userLimit       1（每人限领数量）
```

**多券示例：**

| KEY（couponId） | field | value |
|----------------|-------|-------|
| prs:coupon:10 | issueBeginTime | 20230327 |
| | issueEndTime | 20230501 |
| | totalNum | 100 |
| | userLimit | 1 |
| prs:coupon:20 | issueBeginTime | 20230827 |
| | issueEndTime | 20230901 |
| | totalNum | 200 |
| | userLimit | 2 |

#### 1.2.2.2 用户已领数量记录

**数据结构：Hash**

**选型理由：**

一张券可能被多个用户领取，每个用户的已领数量都需要记录。以优惠券 id 为 key，用户 id 为 field，已领数量为 value，天然符合 Hash 的"一组相关字段"语义。通过 `HINCRBY` 原子累加，既完成计数又能在同一步骤中读取结果，用于校验是否超出限额。

**结构示意：**

```
KEY: prs:user:coupon:{couponId}   →   Hash
    field（userId）    value（已领数量）
    110               1
    120               1
    130               1
    140               2（领了2次）
```

**多券示例：**

| KEY（couponId） | field（userId） | value（count） |
|----------------|----------------|---------------|
| prs:user:coupon:10 | uid:110 | 1 |
| | uid:120 | 1 |
| | uid:130 | 1 |
| | uid:140 | 1 |

#### 1.2.2.3 兑换码序列号范围映射

**数据结构：ZSet（有序集合）**

**选型理由：**

兑换码通过自增序列号生成，需要根据序列号反查其对应的优惠券 id。ZSet 以分数（score）存储序列号最大值，支持 `ZRANGEBYSCORE` 快速范围查询，一次命令即可找到"分数 >= 当前序列号"的最小 member（即对应优惠券）。

**结构示意：**

```
KEY: prs:coupon:range   →   ZSet
    member（couponId）    score（该券兑换码序列号最大值）
    "10"                  100
    "20"                  300
    "30"                  500
```

表示：序列号 1~100 对应券10，101~300 对应券20，301~500 对应券30。

#### 1.2.2.4 兑换码兑换状态

**数据结构：BitMap（String 底层）**

**选型理由：**

兑换状态只有两种（已兑换/未兑换），天然用 1 bit 表示。以序列号为偏移量，`GETBIT` 读取、`SETBIT` 写入均是 O(1) 操作，完全不查数据库。支持 2^32 个 bit 位，与 32 位序列号完美匹配，可覆盖 42 亿兑换码。

**结构示意：**

```
KEY: prs:coupon:code:map   →   BitMap
    offset（序列号）    bit（0=未兑换, 1=已兑换）
    1                   1
    2                   0
    3                   1
```

### 1.2.3 Redis Key 命名规范

采用 **模块前缀 : 业务分类 : 标识符** 的层级结构，各段用 `:` 分隔。

| Key | 数据结构 | 说明 |
|-----|---------|------|
| `prs:coupon:{couponId}` | Hash | 优惠券基础信息缓存 |
| `prs:user:coupon:{couponId}` | Hash | 用户领券数量记录 |
| `prs:coupon:range` | ZSet | 兑换码序列号范围映射 |
| `prs:coupon:code:map` | BitMap | 兑换码兑换状态 |
| `prs:coupon:code:serial` | String | 兑换码序列号自增计数器 |

**命名规则说明：**

- 前缀 `prs` 代表 `promotion`（促销模块），全系统统一前缀，防止 key 冲突
- 第二段区分业务类型（`coupon` / `user` 等）
- 最后一段使用具体 id（`couponId` / `userId`）作为区分标识

代码中通过常量统一定义前缀，避免硬编码：

```java
public class PromotionConstants {
    // 优惠券信息缓存 key 前缀
    public static final String COUPON_CACHE_KEY_PREFIX = "prs:coupon:";
    // 用户领券数量缓存 key 前缀
    public static final String USER_COUPON_CACHE_KEY_PREFIX = "prs:user:coupon:";
    // 兑换码序列号范围 ZSet key
    public static final String COUPON_RANGE_KEY = "prs:coupon:range";
    // 兑换码已使用状态 BitMap key
    public static final String COUPON_CODE_MAP_KEY = "prs:coupon:code:map";
    // 兑换码序列号自增 key
    public static final String COUPON_CODE_SERIAL_KEY = "prs:coupon:code:serial";
}
```

**过期策略：**

| Key | 过期策略 | 说明 |
|-----|---------|------|
| `prs:coupon:{couponId}` | 手动删除 | 暂停发放或优惠券过期时由业务代码删除，不设 TTL |
| `prs:user:coupon:{couponId}` | 跟随优惠券 | 优惠券缓存删除后，此 key 可设 TTL 或随优惠券过期一同清理 |
| `prs:coupon:range` | 不过期 | 全局唯一，存储所有券的序列号范围，长期有效 |
| `prs:coupon:code:map` | 不过期 | 兑换码状态需要永久保存，不能过期 |
| `prs:coupon:code:serial` | 不过期 | 全局自增计数器，重置会导致序列号重复 |

> 设计说明：优惠券缓存不使用 TTL 自动过期，而是由业务代码主动管理。原因：优惠券有效期由业务逻辑控制，不适合交给 Redis 随机过期策略；缓存突然过期会导致请求穿透数据库，影响高并发稳定性；手动删除可以与数据库状态变更保持同步，确保缓存一致性。

### 1.2.4 进阶方案：LUA 脚本原子化校验

上述方案在 Redis 校验阶段仍需多次与 Redis 交互，高并发时会产生多次网络往返。进一步优化方案是**将所有校验逻辑封装成一个 LUA 脚本**，一次请求完成全部校验，保证原子性的同时消除网络往返开销。

使用 LUA 脚本后还有一个额外收益：**无需分布式锁**。Redis 单线程执行 LUA 脚本本身就保证了原子性，并发安全由脚本内部保障。

## 1.3 代码实现

### 1.3.1 优惠券缓存管理

**（1）缓存时机**

优惠券缓存随优惠券生命周期管理，遵循以下规则：

- **立刻发放**时：在 `beginIssue` 方法内同步写入缓存
- **延时发放**时：由定时任务扫描到发放时间后写入缓存
- **暂停发放**时：立即删除缓存（`pauseIssue` 方法内执行）
- **优惠券过期**时：由定时任务扫描后删除缓存

**（2）添加缓存实现**

```java
private final StringRedisTemplate redisTemplate;

@Transactional
@Override
public void beginIssue(CouponIssueFormDTO dto) {
    // 1. 查询优惠券
    Coupon coupon = getById(dto.getId());
    if (coupon == null) {
        throw new BadRequestException("优惠券不存在！");
    }
    // 2. 判断优惠券状态，是否是暂停或待发放
    if (coupon.getStatus() != CouponStatus.DRAFT && coupon.getStatus() != PAUSE) {
        throw new BizIllegalException("优惠券状态错误！");
    }
    // 3. 判断是否是立刻发放
    LocalDateTime issueBeginTime = dto.getIssueBeginTime();
    LocalDateTime now = LocalDateTime.now();
    boolean isBegin = issueBeginTime == null || !issueBeginTime.isAfter(now);
    // 4. 更新优惠券
    Coupon c = BeanUtils.copyBean(dto, Coupon.class);
    if (isBegin) {
        c.setStatus(ISSUING);
        c.setIssueBeginTime(now);
    } else {
        c.setStatus(UN_ISSUE);
    }
    updateById(c);

    // 5. 添加缓存（仅限立刻发放）
    if (isBegin) {
        coupon.setIssueBeginTime(c.getIssueBeginTime());
        coupon.setIssueEndTime(c.getIssueEndTime());
        cacheCouponInfo(coupon);
    }

    // 6. 判断是否需要生成兑换码（领取方式为兑换码 && 原状态为待发放）
    if (coupon.getObtainWay() == ObtainType.ISSUE && coupon.getStatus() == CouponStatus.DRAFT) {
        coupon.setIssueEndTime(c.getIssueEndTime());
        codeService.asyncGenerateCode(coupon);
    }
}

private void cacheCouponInfo(Coupon coupon) {
    Map<String, String> map = new HashMap<>(4);
    map.put("issueBeginTime", String.valueOf(DateUtils.toEpochMilli(coupon.getIssueBeginTime())));
    map.put("issueEndTime", String.valueOf(DateUtils.toEpochMilli(coupon.getIssueEndTime())));
    map.put("totalNum", String.valueOf(coupon.getTotalNum()));
    map.put("userLimit", String.valueOf(coupon.getUserLimit()));
    redisTemplate.opsForHash().putAll(PromotionConstants.COUPON_CACHE_KEY_PREFIX + coupon.getId(), map);
}
```

**（3）移除缓存实现**

```java
@Override
@Transactional
public void pauseIssue(Long id) {
    // 1. 查询旧优惠券
    Coupon coupon = getById(id);
    if (coupon == null) {
        throw new BadRequestException("优惠券不存在");
    }
    // 2. 当前券状态必须是未开始或进行中
    CouponStatus status = coupon.getStatus();
    if (status != UN_ISSUE && status != ISSUING) {
        return;
    }
    // 3. 更新状态（乐观更新，防并发）
    boolean success = lambdaUpdate()
            .set(Coupon::getStatus, PAUSE)
            .eq(Coupon::getId, id)
            .in(Coupon::getStatus, UN_ISSUE, ISSUING)
            .update();
    if (!success) {
        log.error("重复暂停优惠券");
    }
    // 4. 删除缓存
    redisTemplate.delete(PromotionConstants.COUPON_CACHE_KEY_PREFIX + id);
}
```

### 1.3.3 基于 Redis 的资格校验（直接领券）

资格校验通过后，不直接操作数据库，而是发送 MQ 消息：

```java
@Service
@RequiredArgsConstructor
public class UserCouponServiceImpl extends ServiceImpl<UserCouponMapper, UserCoupon>
        implements IUserCouponService {

    private final StringRedisTemplate redisTemplate;
    private final RabbitMqHelper mqHelper;

    @Override
    @Lock(name = "lock:coupon:#{couponId}")
    public void receiveCoupon(Long couponId) {
        // 1. 从 Redis 缓存中查询优惠券
        Coupon coupon = queryCouponByCache(couponId);
        if (coupon == null) {
            throw new BadRequestException("优惠券不存在");
        }
        // 2. 校验发放时间
        LocalDateTime now = LocalDateTime.now();
        if (now.isBefore(coupon.getIssueBeginTime()) || now.isAfter(coupon.getIssueEndTime())) {
            throw new BadRequestException("优惠券发放已经结束或尚未开始");
        }
        // 3. 校验库存
        if (coupon.getIssueNum() >= coupon.getTotalNum()) {
            throw new BadRequestException("优惠券库存不足");
        }
        Long userId = UserContext.getUser();
        // 4. 校验每人限领数量（原子累加，避免并发超领）
        String key = PromotionConstants.USER_COUPON_CACHE_KEY_PREFIX + couponId;
        Long count = redisTemplate.opsForHash().increment(key, userId.toString(), 1);
        if (count > coupon.getUserLimit()) {
            throw new BadRequestException("超出领取数量");
        }
        // 5. 扣减 Redis 缓存库存
        redisTemplate.opsForHash().increment(
                PromotionConstants.COUPON_CACHE_KEY_PREFIX + couponId, "totalNum", -1);
        // 6. 发送 MQ 消息，异步写数据库
        UserCouponDTO uc = new UserCouponDTO();
        uc.setUserId(userId);
        uc.setCouponId(couponId);
        mqHelper.send(MqConstants.Exchange.PROMOTION_EXCHANGE, MqConstants.Key.COUPON_RECEIVE, uc);
    }

    private Coupon queryCouponByCache(Long couponId) {
        String key = PromotionConstants.COUPON_CACHE_KEY_PREFIX + couponId;
        Map<Object, Object> objMap = redisTemplate.opsForHash().entries(key);
        if (objMap.isEmpty()) {
            return null;
        }
        return BeanUtils.mapToBean(objMap, Coupon.class, false, CopyOptions.create());
    }
}
```

### 1.3.4 基于 Redis 的资格校验（兑换码领券）

```java
@Override
@Lock(name = "lock:coupon:#{T(com.tianji.common.utils.UserContext).getUser()}")
public void exchangeCoupon(String code) {
    // 1. 校验并解析兑换码
    long serialNum = CodeUtil.parseCode(code);
    // 2. 校验是否已兑换（BitMap）
    boolean exchanged = codeService.updateExchangeMark(serialNum, true);
    if (exchanged) {
        throw new BizIllegalException("兑换码已经被兑换过了");
    }
    try {
        // 3. 查询兑换码对应的优惠券 id（ZSet 范围查询）
        Long couponId = codeService.exchangeTargetId(serialNum);
        if (couponId == null) {
            throw new BizIllegalException("兑换码不存在！");
        }
        Coupon coupon = queryCouponByCache(couponId);
        // 4. 校验发放时间
        LocalDateTime now = LocalDateTime.now();
        if (now.isAfter(coupon.getIssueEndTime()) || now.isBefore(coupon.getIssueBeginTime())) {
            throw new BizIllegalException("优惠券活动未开始或已经结束");
        }
        // 5. 校验每人限领数量
        Long userId = UserContext.getUser();
        String key = PromotionConstants.USER_COUPON_CACHE_KEY_PREFIX + couponId;
        Long count = redisTemplate.opsForHash().increment(key, userId.toString(), 1);
        if (count > coupon.getUserLimit()) {
            throw new BadRequestException("超出领取数量");
        }
        // 6. 发送 MQ 消息，异步写数据库
        UserCouponDTO uc = new UserCouponDTO();
        uc.setUserId(userId);
        uc.setCouponId(couponId);
        uc.setSerialNum((int) serialNum);
        mqHelper.send(MqConstants.Exchange.PROMOTION_EXCHANGE, MqConstants.Key.COUPON_RECEIVE, uc);
    } catch (Exception e) {
        // 异常时回滚 BitMap 标记，避免兑换码被永久占用
        codeService.updateExchangeMark(serialNum, false);
        throw e;
    }
}
```

根据序列号查找优惠券 id 的实现（利用 ZSet 范围查询）：

```java
@Override
public Long exchangeTargetId(long serialNum) {
    // 查询 score >= serialNum 的第一个 couponId
    // score 存的是该优惠券兑换码序列号的最大值
    Set<String> results = redisTemplate.opsForZSet().rangeByScore(
            COUPON_RANGE_KEY, serialNum, serialNum + 5000, 0L, 1L);
    if (CollUtils.isEmpty(results)) {
        return null;
    }
    return Long.parseLong(results.iterator().next());
}
```

### 1.3.5 MQ 监听器

```java
@RequiredArgsConstructor
@Component
```

```java
public void checkAndCreateUserCoupon(UserCouponDTO uc) {

        throw new BizIllegalException("优惠券库存不足！");
    }

    // 4. 若是兑换码领券，更新兑换码状态为已使用
    if (uc.getSerialNum() != null) {
        codeService.lambdaUpdate()
                .set(ExchangeCode::getUserId, uc.getUserId())
                .set(ExchangeCode::getStatus, ExchangeCodeStatus.USED)
                .eq(ExchangeCode::getId, uc.getSerialNum())
                .update();
    }
}
```

### 1.3.7 进阶：LUA 脚本实现原子校验

将校验逻辑封装为 LUA 脚本，一次网络请求完成全部校验，同时彻底去掉分布式锁。

**加载脚本：**

```java
private static final RedisScript<Long> RECEIVE_COUPON_SCRIPT;
private static final RedisScript<String> EXCHANGE_COUPON_SCRIPT;

static {
    RECEIVE_COUPON_SCRIPT = RedisScript.of(
            new ClassPathResource("lua/receive_coupon.lua"), Long.class);
    EXCHANGE_COUPON_SCRIPT = RedisScript.of(
            new ClassPathResource("lua/exchange_coupon.lua"), String.class);
}
```

```java
@Override
```

**领券 LUA 脚本（`receive_coupon.lua`）：**

```lua
-- KEYS[1]: 优惠券缓存 key（prs:coupon:{couponId}）
-- KEYS[2]: 用户领券记录 key（prs:user:coupon:{couponId}）
-- ARGV[1]: userId

-- 1. 优惠券缓存不存在，说明活动未开始或已结束
if (redis.call('exists', KEYS[1]) == 0) then
    return 1
end
-- 2. 库存不足
if (tonumber(redis.call('hget', KEYS[1], 'totalNum')) <= 0) then
    return 2
end
-- 3. 发放时间已结束
if (tonumber(redis.call('time')[1]) > tonumber(redis.call('hget', KEYS[1], 'issueEndTime'))) then
    return 3
end
-- 4. 超出每人限领数量（先累加，若超出则返回 4）
if (tonumber(redis.call('hget', KEYS[1], 'userLimit'))
        < redis.call('hincrby', KEYS[2], ARGV[1], 1)) then
    return 4
end
-- 5. 扣减库存
redis.call('hincrby', KEYS[1], "totalNum", "-1")
-- 校验全部通过
return 0
```

脚本返回值说明：

| 返回值 | 含义 |
|--------|------|
| 0 | 校验通过 |
| 1 | 优惠券不存在（活动未开始或已结束） |
| 2 | 库存不足 |
| 3 | 发放时间已结束 |
| 4 | 超出每人限领数量 |

**兑换码 LUA 脚本（`exchange_coupon.lua`）：**

```lua
-- KEYS[1]: BitMap key（兑换码状态）
-- KEYS[2]: ZSet key（兑换码序列号范围映射）
-- ARGV[1]: serialNum（序列号）
-- ARGV[2]: serialNum + 5000（ZSet 查询上界）
-- ARGV[3]: userId

-- 1. 校验是否已兑换
if (redis.call('GETBIT', KEYS[1], ARGV[1]) == 1) then
    return "1"
end
-- 2. 查询序列号对应的优惠券 id
local arr = redis.call('ZRANGEBYSCORE', KEYS[2], ARGV[1], ARGV[2], 'LIMIT', 0, 1)
if (#arr == 0) then
    return "2"
end
local cid = arr[1]
local _k1 = "prs:coupon:" .. cid
local _k2 = "prs:user:coupon:" .. cid
-- 3. 优惠券缓存不存在
if (redis.call('EXISTS', _k1) == 0) then
    return "3"
end
-- 4. 发放时间已结束
if (tonumber(redis.call('time')[1]) > tonumber(redis.call('HGET', _k1, 'issueEndTime'))) then
    return "4"
end
-- 5. 超出每人限领数量
if (tonumber(redis.call('HGET', _k1, 'userLimit'))
        < redis.call('HINCRBY', _k2, ARGV[3], 1)) then
    return "5"
end
-- 6. 标记兑换码已使用
redis.call('SETBIT', KEYS[1], ARGV[1], "1")
-- 返回优惠券 id（>= 10 则说明校验通过）
return cid
```

## 1.4 面试要点

**面试官：加锁以后性能比较差，有什么好的办法？**

解决性能问题可以采用 MQ 异步领券方案。核心思路是：**把重业务逻辑从主链路中剥离，利用 MQ 做流量削峰和异步处理，降低数据库并发压力**。

具体来说：
1. 将优惠券关键信息（发放时间、库存、限领数）缓存到 Redis Hash 中
2. 用户请求进入后先读 Redis 完成全部资格校验（不查数据库）
3. 校验通过后通过 MQ 发送异步消息，立即返回"领取成功"
4. MQ 消费者异步完成数据库写入（更新发放数量 + 新增用户券）

---

**面试官：这个方案还有什么可以优化的地方？**

当并发量进一步提高时，Redis 校验阶段仍需多次网络往返（多次 HGET / HINCRBY 命令），可以用 **Redis LUA 脚本**进一步优化。LUA 脚本在 Redis 服务端原子执行，只需一次网络请求就能完成所有校验逻辑，同时天然保证原子性，**连分布式锁都不需要了**。

---

**面试官：Redis 缓存的库存和数据库的库存会不会不一致？**

会有短暂的不一致，但在可接受范围内：

- Redis 库存扣减在前，数据库更新在后（MQ 消费后写入），中间有毫秒级延迟
- 极端情况下 MQ 消费失败，需要有重试机制（RabbitMQ 默认支持）保证最终一致性
- 数据库的 `incrIssueNum` 方法带乐观校验（更新失败说明库存确实不足），作为最后一道防线

整体设计遵循**最终一致性**原则，符合高并发场景的工程实践。
