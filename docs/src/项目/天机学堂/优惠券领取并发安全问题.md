
## 1.1 需求分析

领取优惠券功能涉及四个核心接口：

- **查询优惠券列表**：查询发放中且手动领取的优惠券，标注"可领取/已领取/已抢完"状态
- **领取优惠券**：校验库存、限领数量后，新增用户券记录并更新已发放数量
- **查询我的优惠券**：分页查询当前用户已领取的优惠券，支持按状态过滤
- **兑换优惠券**：输入兑换码，校验后领取优惠券（BitMap 标记兑换状态）

**数据库设计核心：** `user_coupon` 表记录用户与券的领取关系，包含用户 id、优惠券 id、有效期、使用状态等字段。

---

## 1.2 方案设计

### 1.2.1 领券核心流程

领取优惠券需要依次完成以下校验：

1. 校验优惠券是否存在
2. 校验发放时间是否有效
3. 校验库存是否充足（`issue_num < total_num`）
4. 校验每人限领数量
5. 更新已发放数量（`issue_num + 1`）
6. 新增用户券记录

其中，步骤 3（库存校验）和步骤 4（限领校验）涉及共享资源的并发读写，是高并发场景下的安全痛点。

### 1.2.2 兑换码兑换流程

与手动领券类似，额外增加兑换码校验：

1. 解析兑换码格式，获取兑换码 id（serialNum）
2. 通过 Redis BitMap 校验是否已兑换（`SETBIT key serialNum 1`，**直接set**，通过返回值判断是否已兑换）
3. 查询兑换码对应的优惠券，校验是否过期
4. 执行后续领券逻辑（同手动领券）
5. 异常时回滚 BitMap 标记（`SETBIT key serialNum 0`）

## 1.3 并发安全问题

领券过程涉及多次数据库读写（查询库存 → 判断 → 更新），多线程并发时极易出现安全问题。以下逐一分析各场景及解决方案。

### 1.3.1 超卖问题——乐观锁

**问题现象：** 优惠券库存 100，高并发测试后发放了 109 张。

**原因分析：** 库存判断采用"先查询，再判断，再更新"的三步走，三步不具备原子性。多线程并行时，多个线程同时查询到库存充足，一起执行更新，导致超卖。

**解决方案——乐观锁：**

乐观锁在更新时通过 `WHERE` 条件做 CAS 校验，判断数据是否发生变化。针对库存场景，无需判断 `issue_num` 是否与原来一致，只需判断 `issue_num < total_num` 即可：

```java
// Mapper 层：更新库存时附加 WHERE 条件
@Update("UPDATE coupon SET issue_num = issue_num + 1 WHERE id = #{couponId} AND issue_num < total_num")
int incrIssueNum(@Param("couponId") Long couponId);
```

```java
// Service 层：根据返回值判断更新是否成功
int r = couponMapper.incrIssueNum(coupon.getId());
if (r == 0) {
    throw new BizIllegalException("优惠券库存不足");
}
```

> 若 `WHERE` 条件不成立，更新返回 0，此时抛出异常触发事务回滚。

**悲观锁 vs 乐观锁对比：**

- **悲观锁（Synchronized/ReentrantLock）**：独占资源，线程串行执行。优点是安全性高，缺点是性能差。
- **乐观锁（CAS）**：不加锁，更新时判断数据是否被修改。优点是性能好，缺点是并发高时成功率低。
- **乐观锁改进（库存判断）**：`WHERE issue_num < total_num`。既保证了性能好、成功率高，又无需版本号机制，是库存场景的最佳实践。

---

### 1.3.2 锁失效问题——锁对象必须同一

**问题场景：** 用户限领数量的校验逻辑同样存在并发安全问题（查询已领数量 → 判断是否超限 → 新增用户券），需用**悲观锁**解决。

**锁设计：** 锁的粒度是"单个用户"，使用 `synchronized(userId)` 代码块，这样**同一用户并发时被串行**，**不同用户互不影响**。

**问题现象：** 加了锁，但并发测试发现锁未生效，用户仍能超领。

**原因分析：** 锁对象使用了 `userId.toString()`，而 `Long.toString()` 源码是 `new String(...)`，导致同一 userId 每次 toString 都生成不同对象，也就是多把不同的锁。

**解决方案——`String.intern()`：**

`String.intern()` 获取字符串在常量池中的唯一引用，保证 equals 相等的字符串 == 也相等：

```java
synchronized(userId.toString().intern()) {
    checkAndCreateUserCoupon(coupon, userId, null);
}
```

> 只要两个字符串内容相同，`intern()` 返回的就是同一个对象，确保锁生效。

### 1.3.3 事务边界问题——锁与事务的先后顺序

**问题现象：** 锁生效后，用户限领数量的并发安全问题依然存在，用户仍然超领。

**原因分析：** 领券方法加了 `@Transactional` 事务，而 `synchronized` 锁在方法内部。执行顺序为：

```
开启事务 → 获取锁 → 执行业务 → 释放锁 → 提交事务
```

线程 1 释放锁后，线程 2 立刻获取锁执行。但线程 1 的事务尚未提交，线程 2 读取不到未提交的领券记录，导致判断"未超限"，从而超领。

**解决方案——调整锁与事务的边界：**

将锁放到事务外层，保证"先获取锁，再开启事务；先提交事务，再释放锁"：

```java
@Override
public void receiveCoupon(Long couponId) {
    // ... 前置校验（查券、查时间、查库存）
    Long userId = UserContext.getUser();
    // 锁在事务外：先加锁
    synchronized(userId.toString().intern()) {
        // 再进入事务方法
        checkAndCreateUserCoupon(coupon, userId, null);
    }
}

@Transactional
public void checkAndCreateUserCoupon(Coupon coupon, Long userId, Integer serialNum) {
    // 1. 校验限领数量
    Integer count = lambdaQuery()
            .eq(UserCoupon::getUserId, userId)
            .eq(UserCoupon::getCouponId, coupon.getId())
            .count();
    if (count != null && count >= coupon.getUserLimit()) {
        throw new BadRequestException("超出领取数量");
    }
    // 2. 更新库存（乐观锁）
    int r = couponMapper.incrIssueNum(coupon.getId());
    if (r == 0) {
        throw new BizIllegalException("优惠券库存不足");
    }
    // 3. 新增用户券
    saveUserCoupon(coupon, userId);
}
```

> 由于 `@Transactional` 需要 `public` 方法且被 Spring 代理，因此将事务方法抽取到 Service 接口中，确保 Spring AOP 能正确拦截。

### 1.3.4 事务失效问题——非事务方法调用事务方法

**问题现象：** 调整锁与事务边界后，并发安全解决了，但事务失效了——抛异常后库存和用户券未回滚。

**原因分析：** 事务注解从 `receiveCoupon` 移到了 `checkAndCreateUserCoupon`，而 `receiveCoupon` 内部通过 `this` 直接调用 `checkAndCreateUserCoupon`，走的是原始对象而非 Spring 代理对象，事务 AOP 未生效。

**Spring 事务失效的常见原因：**

- **方法非 public**：Spring AOP 基于代理，非 public 方法无法被拦截
- **非事务方法调用事务方法**：`this.` 调用走原始对象，绕过代理，事务不生效
- **异常被捕获**：事务依赖异常感知回滚，捕获后 Spring 无法感知
- **异常类型不对**：默认只回滚 RuntimeException，IOException 等需指定 `rollbackFor`
- **传播行为错误**：`REQUIRES_NEW` 会创建独立子事务，外部异常不影响子事务
- **Bean 未被 Spring 管理**：缺少 `@Service` 等注解，或对象非 Spring 创建

**解决方案——获取 Spring 代理对象：**

借助 AspectJ 暴露代理对象，通过代理对象调用事务方法：

1. 引入 AspectJ 依赖：

```xml
<dependency>
    <groupId>org.aspectj</groupId>
    <artifactId>aspectjweaver</artifactId>
</dependency>
```

2. 启动类添加注解暴露代理对象：

```java
@EnableAspectJAutoProxy(exposeProxy = true)
```

3. 通过 `AopContext` 获取代理对象调用事务方法：

```java
@Override
public void receiveCoupon(Long couponId) {
    // ... 前置校验
    Long userId = UserContext.getUser();
    synchronized(userId.toString().intern()) {
        // 获取代理对象，确保事务 AOP 生效
        IUserCouponService proxy = (IUserCouponService) AopContext.currentProxy();
        proxy.checkAndCreateUserCoupon(coupon, userId, null);
    }
}
```

## 1.4 面试要点

---

**面试官：如何解决优惠券的超发问题？**

超发问题源于多线程并发访问共享资源（库存），核心手段是加锁。可采用悲观锁或乐观锁：

- **悲观锁**：`Synchronized` 或 `ReentrantLock`，线程串行执行，安全性高但性能差，适合并发量不大的场景。
- **乐观锁**：不加锁，更新时通过 `WHERE` 条件判断数据是否被修改。针对库存场景，无需用版本号或 CAS 判断原值，直接在 `WHERE` 中加上 `issue_num < total_num` 的条件即可。这样只要库存充足，多个线程都能成功更新；库存不足时更新失败，兼具线程安全和并发性能，同时避免了乐观锁失败率高的问题。

---

**面试官：Spring 事务失效的情况碰到过吗？**

事务失效的原因很多，常见有以下几类：

- 事务方法不是 `public` 修饰的
- 非事务方法调用事务方法（`this.` 调用绕过代理）
- 事务方法内部捕获了异常，Spring 无法感知
- 抛出的异常类型不在默认回滚范围内（如 `IOException`，需指定 `rollbackFor = Exception.class`）
- 事务传播行为使用不当（如 `REQUIRES_NEW` 导致子事务独立）
- Bean 没有被 Spring 管理

在实际项目中确实遇到过：优惠券业务中，手动领取和兑换码兑换有公共逻辑，于是抽取为一个共享方法并将 `@Transactional` 注解放到了抽取的方法上。两个业务方都是非事务方法直接调用这个公共事务方法，导致事务失效。表现为领取失败时库存扣减未回滚，最终发放数量与设定数量对不上。排查后通过 AspectJ 暴露代理对象，用 `AopContext.currentProxy()` 获取代理再调用，解决了问题。

---

**面试官：在开发中碰到过什么疑难问题，最后是怎么解决的？**

优惠券功能中，库存校验和用户限领数量校验都需要先查询再判断，高并发下存在线程安全问题。

库存更新可以直接用乐观锁（`WHERE issue_num < total_num`），但限领数量校验是新增操作，无法使用乐观锁，只能加悲观锁。为减少锁粒度、提升并发性能，锁对象设计为 `userId`，这样同一用户串行、不同用户并行。

但测试发现加了锁后用户仍然超领。排查后发现锁本身没问题，真正的原因是**事务边界与锁边界不匹配**：原代码先开启事务再获取锁，业务结束后先释放锁再提交事务。线程 1 释放锁后，线程 2 获取锁执行，但线程 1 的事务尚未提交，线程 2 读不到未提交的领券记录，导致判断错误。

最终方案是将锁调整到事务外层：先获取锁，再进入事务方法；事务方法内完成所有数据库操作并提交后，锁才释放。同时，由于事务方法被非事务方法通过 `this.` 调用导致事务失效，又通过 `AspectJ` 暴露代理对象解决。最终并发安全和事务回滚都得以保障。
